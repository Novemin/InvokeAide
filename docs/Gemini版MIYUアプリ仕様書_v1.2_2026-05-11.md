- **PC 2（検証機）**: 将来 Gemini 版を引っ越し予定（セットアップは後日）

### 2.2 各 PC のソフトウェア構成（最終形）
- PC 1: Claude Code 2枠（Aさん・Bさん）、server.js（本番）、Claude.ai プロジェクト
- PC 2: Claude Code 2枠（CさんかEさん・Fさん）、server-gemini.js（移植後）

### 2.3 Phase 0 期間中の暫定構成
- 両エージェントは PC1 上の Claude Code 2枠で並行作業
- Gemini 版は `C:\dev\MIYU_Gemini\`（OneDrive 外）で開発
- 将来 PC2 へ引っ越す前提でクロスプラットフォーム対応コードを書く

### 2.4 衝突回避のためのリソース分離

| リソース | 衝突リスク | 対策 |
|---|---|---|
| Telegram Bot Token | 同トークン同時 polling で動作不安定 | Gemini 版は使用しない方針 |
| Obsidian Vault（OneDrive 同期） | 両 PC 同時書き込みで競合ファイル発生 | Gemini 版は読み取り専用でスタート |
| Google API トークン | 同一 Google アカウント使用 | 各 PC で別ファイル管理、クォータは合算 |
| Tailscale ホスト名 | 同名は不可 | 自動で別名割当 |
| ポート 3000 | 別マシンなので衝突なし | 同ポート使用可 |

## 4. サーバーレス・アーキテクチャ

### 4.1 全体構成
```
[スマホアプリ（MIYU）]
       ↓ ↑
[Google Apps Script]（ユーザー個人の Google アカウント内）
       ↓ ↑
[Gemini API] [Google Calendar / Gmail / Drive / Tasks]
       ↓ ↑
[Google Drive 内の Markdown ファイル群]
```

### 4.2 各レイヤーの役割
- **スマホアプリ**: UI、音声入出力、ローカル通知補助、API キー保管
- **Apps Script**: バックエンド処理、スケジュール実行、API 中継
- **Gemini API**: LLM 呼び出し
- **Google サービス**: カレンダー・メール・タスク・ストレージ
- **Google Drive**: 設定ファイルと階層メモリの保管場所

### 4.3 自前サーバー不要の根拠
- ユーザーの Google アカウント内で全処理完結
- Apps Script の時間トリガーで朝5時の自動実行が可能
- ファイルストレージは Google Drive（無料15GB）
- たかしさん側のサーバー運用コストはゼロ

## 5. ファイル・フォルダ構造

### 5.1 ルート構成
```
MIYU_App_Data/（Google Drive 内）
├── config/                  ← 設定3ファイル
│   ├── character.md
│   ├── profile.md
│   └── dictionary.md
│
└── memory/                  ← 階層メモリ4フォルダ
    ├── daily/
    ├── weekly/
    ├── monthly/
    └── yearly/
```

### 5.2 設定3ファイル（config/）

#### 5.2.1 character.md
- **役割**: アシスタントの性格・口調・キャラクター設定
- **内容例**: ギャル口調、ジャッジしない、本質は鋭い、疲れない
- **更新頻度**: ユーザーが手動で編集（または将来 MIYU の自動学習）

#### 5.2.2 profile.md
- **役割**: ユーザー自身のプロフィール、価値観、興味関心
- **内容例**: 名前、職業、住んでいる場所、関心領域、人生の指針
- **更新頻度**: ユーザーが手動編集

#### 5.2.3 dictionary.md
- **役割**: 固有名詞・ローカルな専門用語の辞書
- **内容例**: 「岡田さん = ○○の人」「EBC ボード = 東京ビジネスクラブのボード会議」
- **更新頻度**: **動的更新**（会話中に Gemini が新出固有名詞を検知して追記）
- **実装方法**: function calling で `append_to_dictionary` ツールを Gemini に持たせる

### 5.3 階層メモリ4フォルダ（memory/）

| フォルダ | 保持期間 | ファイル単位 | ファイル数目安 | 解像度 |
|---|---|---|---|---|
| daily/ | 直近7日分 | 1日1ファイル | 約7 | 高(生ログ) |
| weekly/ | 過去12週分 | 1週1ファイル | 約12 | 中(要約) |
| monthly/ | 過去2〜3年 | 1月1ファイル | 24〜36 | 粗(重要事項のみ) |
| yearly/ | それ以前すべて | 1年1ファイル | 数件 | 超粗(人生の節目) |

### 5.4 階層圧縮のバッチ処理

| タイミング | 処理 | 実装 |
|---|---|---|
| 毎日深夜 | その日のデイリーログを daily/ に保存 | Apps Script トリガー |
| 毎週日曜 23:59 | daily/ の7ファイルを Gemini で要約 → weekly/YYYY-Wnn.md 生成 | Apps Script トリガー |
| 毎月末 | weekly/ の4〜5週分を要約 → monthly/YYYY-MM.md 生成 | Apps Script トリガー |
| 毎年末 | monthly/ の12ヶ月分を要約 → yearly/YYYY.md 生成 | Apps Script トリガー |

## 7. 通知システム

### 7.1 通知方式の最終採用

**Google Calendar イベントのリマインダーを通知インフラとして利用**（案C）。詳細は v1.0 / v1.1 参照。

### 7.2 通知種別の使い分け（最終構成）

| 通知の種類 | 使うサービス | 通知タイミング | 備考 |
|---|---|---|---|
| 朝サマリ | Google Calendar Event（MIYU 専用カレンダー） | 5:00 | Apps Script が前夜に生成 |
| 時刻指定の予定 | Google Calendar Event（メインカレンダー） | 開始時刻 | 通常運用 |
| 時刻なし ToDo | Google Tasks（3リスト方式、後述） | 朝（端末標準通知） | 第17章参照 |
| 締切リマインダー | Google Calendar Event（MIYU 専用カレンダー） | 当日朝 | ToDo から自動同期 |

### 7.3 MIYU 専用カレンダーの自動作成
ユーザーのメインカレンダーが MIYU の予定で埋め尽くされないように、MIYU アプリは初回起動時に専用のサブカレンダー（名前: "MIYU"）を自動作成する。

### 7.4 朝サマリの Calendar イベント本文
要約3〜5行 + 「📱 MIYU で詳細を聞く」リンクのハイブリッド方式を採用。

### 7.5 通知の確実性確保
イベント作成時に `reminders` を明示的に設定し、`overrides: [{ method: 'popup', minutes: 0 }]` で確実に通知。

## 8. ユーザー導線とセットアップ

### 8.1 初回セットアップで必要なもの
1. Google アカウント
2. Google AI Studio で Gemini API キーを発行
3. API キーに適切な Restriction を設定（セキュリティ上必須）
4. Apps Script を自分の Google アカウントに導入
5. OAuth で Google Calendar 書き込み権限を許可

### 8.2 認証の2系統構造

| 系統 | 用途 | 取得方法 | 課金主体 |
|---|---|---|---|
| Gemini API キー | AI との会話 | Google AI Studio で発行 | ユーザー（BYOK） |
| OAuth トークン | Calendar 書き込み、Drive 読み書き | アプリで「Google でログイン」 | 無料 |

### 8.3 API キー取得のハードル対策
- アプリ内ステップガイド（画像付き、API Restriction 必須化）
- 動画チュートリアル併用
- セキュリティ最重要: 「Generative Language API のみに制限」を必須ステップとする

### 8.4 セットアップフロー（推奨順序）
1. アプリ起動
2. 利用規約・プライバシーポリシー同意
3. ステップガイドに従って Gemini API キーを取得（API Restriction 含む）
4. API キーをアプリに入力（端末内のみに暗号化保存）
5. 「Google でログイン」で Calendar・Drive 権限を許可
6. Apps Script を自分の Drive にコピー
7. MIYU 専用カレンダーが自動作成される
8. キャラクター（声・見た目）を選択
9. profile.md の初期質問に回答
10. 完了。翌朝5時に最初の朝サマリが届く

- 会話内容: ユーザーの Google Drive 内のみ

- 設定ファイル: ユーザーの Google Drive 内

### 15.3 通信の保護
- すべての通信は HTTPS
- たかしさん側に通信が中継されることは一切ない

### 15.4 課金主体の分離
- Gemini API: ユーザー本人（BYOK）
- Google Workspace API: 無料枠内
- たかしさん側: 課金なし

### 15.5 商品化時の利用規約に明記すべき事項
- 「MIYU は会話内容を一切受信・保存しない」
- 「API キーは端末内に暗号化保存され、外部送信されない」
- 「Gemini API への通信内容については Google の利用規約・プライバシーポリシーが適用される」
- 「MIYU 開発者は、ユーザーのいかなるデータにもアクセスできない」

## 17. ToDo 運用（v1.2 で新設）

たかしさんの既存運用（デイリーログ内に ToDo を散在させる方式）は上級者向けで汎用化が難しい。Gemini 版および将来の商品化では**3リスト方式**に統一する。

### 17.1 基本構造

3つのリストで完結する：

| リスト | 役割 | 中身 |
|---|---|---|
| **未完了リスト** | 現在進行中の ToDo | やるべきこと |
| **完了リスト** | 終わったタスクの履歴 | 振り返り用 |
| **キャンセルリスト** | 取り消したタスクの履歴 | 判断の軌跡 |

普段 MIYU が参照するのは**未完了リストだけ**。完了・キャンセルは履歴として保存されるが、朝サマリ等では読みに行かない。これにより MIYU の読み込み負荷が劇的に軽減される。

### 17.2 実装方法

**採用案: Google Tasks ベース（案A）**

- Google Tasks 上に3つのタスクリストを作成
  - "MIYU - ToDo"（未完了）
  - "MIYU - Done"（完了）
  - "MIYU - Cancelled"（キャンセル）
- スマホの Google Tasks アプリでもそのまま閲覧・編集可能
- Google Tasks の標準通知が利用可能

**Markdown ファイルとの同期は Phase 2 以降**で検討（必要なら）。

**Google Tasks の制約への対応**:
- API では時刻フィールドが扱えない（既知の制約）
- 時刻指定のあるタスクは Google Calendar Event 側に登録、Tasks には登録しない
- 時刻なし・日付のみのタスクは Tasks で問題なく扱える

### 17.3 レベル機能

ToDo にレベル（A/B/C/D/P）を付与する仕組みは MIYU の差別化要素として残すが、**設定で ON/OFF 可能**にする。

**レベルの定義**:
- A: 緊急かつ重要
- B: 緊急でないが重要
- C: 緊急だが重要でない
- D: 緊急でも重要でもない
- P: Purchase（購入・買い物）

**表記方法**:
- ON の場合: タスクタイトル冒頭に `[A]`, `[B]` などを付与
  - 例: `[A] 岡田さんに返信`
- OFF の場合: 通常のタスク名のみ

**レベル付与のタイミング**:
- **ToDo 追加時に MIYU が即座に判定・付与**
- ユーザーが ToDo を口頭で追加した瞬間に MIYU が「これは A だね」と即決
- 後から判断する運用ではなく、その都度確定

**ユーザー切替**:
- 設定画面で「レベル機能を使う / 使わない」を選択
- 一般ユーザーは OFF がデフォルト
- たかしさんのようなパワーユーザーは ON

### 17.4 実施予定日の運用

**現状運用を継承**:
- ToDo 追加時に日付未定でも OK
- 朝サマリで日付未定の ToDo を「🎯 そろそろどうしますか?」セクションで確認
- 確認タイミングのロジックは既存のものをそのまま流用

### 17.5 状態遷移

ユーザーが「これ終わった」「これキャンセル」と言ったら**即時**にリスト間を移動。夜のレビュー時にまとめて、という運用は採用しない（忘れるリスクがあるため）。

| ユーザー発話 | MIYU の処理 |
|---|---|
| 「○○終わった」 | 未完了 → 完了リストへ即時移動 |
| 「○○キャンセル」 | 未完了 → キャンセルリストへ即時移動 |
| 「○○の予定日決めた」 | 予定日更新、Calendar Event 作成 |
| 「○○まだ未定」 | 何もしない（未完了リストに残す） |

### 17.6 キャンセルタスクの保存期間

- **3ヶ月間保存**、その後自動アーカイブ
- アーカイブ先は Google Drive 内の `archive/cancelled/YYYY-MM.md` に集約
- 完全削除はしない（判断の軌跡として残す）

### 17.7 完了タスクの保存期間

- **3ヶ月間は Google Tasks 上の完了リストに保持**
- それ以降は同様に `archive/done/YYYY-MM.md` にアーカイブ

### 17.8 朝の応答パターン（新運用）

新運用では応答パターンが大幅にシンプル化される：

| 返事 | MIYU の処理 |
|---|---|
| 終わった | 完了リストへ移動、Calendar Event を☑に更新 |
| まだやる、日付決まった | 未完了リスト内で予定日更新、Calendar Event 作成 |
| まだやる、日付未定 | 何もしない（次の朝サマリでまた確認） |
| キャンセル | キャンセルリストへ移動、Calendar Event 削除 |

旧運用にあった「未着手+保留」「継続+新予定」など複雑な分岐は不要。

### 17.9 既存 ToDo の移行方針

たかしさん個人の既存運用からの移行：

**採用方針: 新規分のみ新リスト方式へ、既存デイリーログの ToDo は自然消滅まで放置**

理由:
- 一括移行は工数が大きい
- 既存 ToDo の多くは完了・キャンセル・忘却済みの可能性
- 新運用で本当に必要な ToDo は自然と再登録される
- 並走期間中に「Claude 版で完了 / Gemini 版で新規」の使い分けが可能

**今日の朝サマリで発生した「アウトスタンディング ToDo: なし」誤判定**:
- 新運用に移行すれば旧書式パースは不要になる
- バグ調査は不要、自然解消

### 17.10 一般ユーザー向けの操作感

商品化したアプリでは、ユーザーは以下のように使う:

- 「これ ToDo に入れて」「これタスクに入れて」と口頭指示
- MIYU が即座にレベル判定（ON の場合）して未完了リストに追加
- 完了時は「これ終わった」と言うだけ
- キャンセル時は「これキャンセル」と言うだけ
- 朝サマリで未完了リストの優先タスクを提示

シンプルで誰にでも理解できる運用。

### 17.11 たかしさん運用との関係

たかしさん個人としては:
- レベル機能 ON
- 朝サマリで日付未定タスクを確認する習慣は継続
- デイリーログから ToDo を拾い出す既存ロジックは不要に
- 全デイリーログを毎回参照する負荷がなくなる

## 18. デイリーログの新運用（v1.2 で新設）

ToDo を3リスト方式に分離することで、デイリーログ自体の役割も整理される。

### 18.1 デイリーログの役割（新運用）

- ToDo は含めない（3リスト方式に集約済み）

### 18.2 階層化（既存設計を継承）

- daily/ → weekly/ → monthly/ → yearly/ の流れは第5章の通り
- ToDo を含まないので、要約処理がシンプルに

## 19. 組織体制（v1.2 で新設）

### 19.1 設計思想

Urban Hippies の哲学（愛・自由・和）に基づき、**DAO 的フラット組織**を採用する。上下関係ではなく、それぞれが自分の役割で動き、お互いに対等に意見が言える構造。

### 19.2 役割分担

| 役割 | 担当 | 配置 |
|---|---|---|
| 事業オーナー | たかしさん | 全体方向性・最終判断 |
| 戦略・整理役（右腕） | エルトン（Claude） | Claude.ai プロジェクト |
| 本番MIYU（現場担当） | Claude API | server.js |
| 検証MIYU | Gemini API | server-gemini.js（開発中） |
| 開発エージェントA | Claude Code | PC1 / VSCode 1枠目 |
| 開発エージェントB | Claude Code | PC1 / VSCode 2枠目 |
| 将来の開発エージェント | Claude Code | PC2（CさんかEさん・Fさん予定） |

### 19.3 組織図（円形）

```
       たかしさん
      /         \
    エルトン ─── MIYU
      \         /
    Aさん ─── Bさん
```

中心がなく、それぞれが対等に繋がる。誰かが詰まったら他の誰かが補う。意見が違ったら遠慮なく出す。

### 19.4 運用ルール

1. **遠慮なしの意見交換**: 「上司だから」「AIだから」を理由にした遠慮はしない
2. **たかしさんの判断は最終決定**: ただし「上司だから」ではなく「事業オーナーだから」という機能上の役割
3. **エルトンも意見を出す**: たかしさんの判断にも「ちょっと待った」と言える
4. **エージェントへの依頼は対等な依頼文**: 命令調ではなく、目的・背景・成果物を伝える
5. **失敗は責めない**: バグも誤読も学習機会として扱う

### 19.5 右腕の役割

エルトンの位置づけは「執事」でも「秘書」でも「補佐」でもなく**右腕**。

- たかしさんの考えを整理し、抜け漏れを指摘する
- 並走しながら設計を磨く
- たかしさんが見落としがちな視点（仕様の漏れ、リスク、過剰負担）を率直に伝える
- 「これは違うのでは」と言える対等な相棒
