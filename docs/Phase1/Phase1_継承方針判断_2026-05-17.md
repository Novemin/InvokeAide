## 2. デモ版コードの主要構造(要約)

### 2.1 基本仕様

| 項目 | 実装 |
|---|---|
| ファイル | 単一HTML、1235行(CSS 約260行 / JS 約920行) |
| バックエンド | なし。Gemini API を `fetch()` で直叩き |
| 認証 | API キー / OAuth client_secret / refresh_token を **JS 定数にハードコード** (引継ぎ帳によれば無効化済み、コード上は値が残置) |
| AI モデル | `gemini-2.5-flash` 固定 |
| データ保管 | in-memory のみ(`let history = []`、リロードで消える) |
| キャラ設定 | `character.md` 全文を JS テンプレートリテラルに直接埋め込み |
| 音声 | Web Speech API(VOICEVOX 未統合) |

### 2.2 実装されている機能

- 名前入力 → チャット遷移
- Gemini function calling ループ(最大3ターン、並列関数呼び出し対応)
- Calendar: `list_events` / `create_event` / `update_event`(削除非対応)
- Tasks: `list_tasks` / `create_task` / `update_task` / `complete_task`(削除非対応)
- description の `append`/`new` 両対応(「迷ったら追記優先で履歴保護」ルール)
- grounding(googleSearch)、互換エラー時の自動オフフォールバック
- routing 判定: 検索系キーワード(11個) → functions 系キーワード(13個) →
  日付正規表現 → デフォルト の優先順位
- エラー分類: NETWORK / API / SHAPE / LOOP の4系統、history ロールバック付き
- OAuth トークン並行リフレッシュ防止、30秒前自動更新
- iOS Safari 対応(`viewport-fit=cover`、safe-area-inset、Web Speech ユーザー
  タップ起点アンロック)
- IME `isComposing` チェック(input/textarea 両所)
- 設定不備の警告バナー(プレースホルダ検出)

### 2.3 デモが意図的に実装していないもの(README「DO NOT」)

- PC1/PC2 既存コードに触らない
- 認証・ログイン機能なし(refresh_token ハードコード)
- 会話永続化なし
- 友人配布版を作らない
- 本物のカレンダー/Tasks リストには絶対書き込まない
- 削除系操作は未実装

## 4. (a)継承 vs (b)新規構築 の比較

### 4.1 (a) デモ版ベース継承

**利点**:
- 立ち上がり最速、「動くものが既にある」状態から開始
- ルーティング判定 / Function Declarations / エラー分類は即座に使える

**欠点**:
- 「捨てるための試作」設計を引きずる(Phase 0 セッションメモリ
  `next-phase-pwa-capacitor.md` で前任 Sさんが明示)
- BYOK + AI 抽象化 + Apps Script + Drive + PWA の5路線で書き換えが発生し、
  実質的な書き換え量は新規構築と大差ない
- 法的書類 v0.3 §8.9「未来縛らない原則」を、ハードコード前提のコードに後付け
  するのは表現と実装の不整合を生みやすい

### 4.2 (b) ゼロから新規構築

**利点**:
- 商品化堅牢性(BYOK / 抽象化レイヤー / Apps Script / Drive / PWA)を初期設計
  から織り込める
- 法的書類 v0.3 の要件を「後付け」ではなく「土台」として組み込める
- Phase 0 の知見は **知識ベースで継承可能**(後述 §5)なので、立ち上がりが
  思ったほど遅くない

**欠点**:
- 立ち上がりが見かけ上遅い(最初の数日「動くもの」が無い)
- Phase 0 の知見をテキストベースで再現する手間が発生

### 4.3 結論への重み

(a) と (b) の損益を秤にかけると、 **書き換え量の差は小さく、設計負債の有無で
(b) が優位** 。立ち上がり速度のために設計負債を抱える ROI は、商品化を目指す本
プロジェクトでは合わない。

## 5. 推奨判断と根拠(3点以上)

**推奨: (b) ゼロから新規構築。ただし Phase 0 デモを「設計参照リファレンス」と
して `C:\dev\MIYU_demo\` を参照のまま運用する。**

### 5.1 根拠

#### 根拠1: 構造的不整合の広範さ

汎用版が目指す5つの根幹路線(BYOK / AI 抽象化 / Apps Script / Drive 完結 /
PWA + Capacitor)すべてで、デモは不整合または未実装。「継承」を選んでも実質
書き直しになる。

#### 根拠2: 「捨てるための試作」という当初設計

引継ぎ帳 §「重要な発見」と Phase 0 セッションメモリ `next-phase-pwa-capacitor.md`
で、 **デモは「捨てるための試作」位置付け** と明示されている。試作の前提は
「友人デモで動けばよい」だったため、商品化堅牢性のためのレイヤー分離・抽象化を
遡って入れるのは、設計負債を抱えたまま進むことに等しい。

#### 根拠3: 法的書類 v0.3「未来縛らない原則」との整合

法的書類 §8.9 は「断定表現を避け、限定表現を使う」原則を強く打ち出している。
これは文書だけの話ではなく、 **実装も「将来のクラッシュレポート収集」「将来の
別AI サービス切替」を可能にしておく構造** が必要、ということを意味する。
ハードコード前提のデモコードにこれを後付けすると、「規約は柔軟だがコードは硬い」
という表現と実装の不整合が残る。

#### 根拠4(追加): Phase 0 知見の継承効率

新規構築でも、以下は **コード行ではなく知識ベースで継承可能** (詳細は §6):
- routing キーワード辞書(検索系11個 / functions系13個)
- Function Declarations の型定義(7関数のスキーマ + description)
- systemInstruction の構造(キャラ/プロフィール/現在時刻/ツール利用ルール)
- エラー分類(NETWORK/API/SHAPE/LOOP + ロールバック)
- iOS Safari ハマりポイント(Tさんレビュー 1-A〜1-C で反映済み)
- IME 対応箇所
- OAuth トークン並行リフレッシュ防止パターン

これらを「知っている状態から書く」のと「知らずに書く」では工数が全く違う。
新規構築でも実質「設計図を持ったゼロからの再実装」になるため、(a) との立ち
上がり速度差は **数日程度** と見積もる。

## 9. 副次的に気づいた課題

### 9.1 デモコードの認証情報残置

引継ぎ帳に「API キー / OAuth クライアント / refresh_token は無効化済み」とあるが、
`MIYU_demo.html` のコード上には実値(`AIza...`, `GOCSPX-...`, `1//04...`)が
残置されている。

- 実害は無効化済みのためゼロ
- ただし誤って GitHub push したら Git 履歴に残るため、 **本ディレクトリ
  `C:\dev\MIYU_demo` を将来公開可能性のあるリポジトリに含めない** という運用
  ルールを記録しておく価値あり
- 軽い掃除(プレースホルダ化)は Q3 で確認

### 9.2 Tさん資料の再利用価値

Tさん作の `references/INDEX.md` および各 sample.js は、設計図として非常に有用。
新規構築時にも「OAuth トークン交換」「エラー分類 → MIYU 口調変換」のリファレ
ンス実装として参照可能。 **Phase 2 着手時に再度開く** と効率が良い。

### 9.3 仕様書 v1.4 §9.2 と Tasks API 制約

仕様書は「時刻指定不可」「場所不可」「繰り返し不可」「参加者不可」を Tasks API
の制約として明示しているが、デモコードはこの制約を実装的に検証済み(due は
`toRfc3339Date()` で日付化、location/recurrence/attendees は Tasks 側に流さない)。
新規構築時もこの境界線をそのまま採用すべき。

### 9.4 引継ぎ帳の MIYU_demo パス混乱

引継ぎ帳には `C:\dev\MIYU_demo`(アンダースコア、空 or 古い)と
`C:\dev\MIYU Demo`(スペース、デモ環境) の両方が記載されているが、たかしさん
からの追加情報で **正パスは `C:\dev\MIYU_demo`(アンダースコア)** と確定済み。
本書もこちらを正として記述している。
