## 1. フロントエンドフレームワーク補足(v0.2 大幅補強)

### 1.1 推奨: Vue 3 + Vite(変更なし)

ただし **「動作の軽さ」観点での定量比較を補足** 。

### 1.2 動作の軽さ — 定量比較

#### 1.2.1 バンドルサイズ(2026年時点の公表値、minified + gzip)

| 項目 | Vue 3 | React 18 | 差 |
|---|---|---|---|
| コアランタイム | 約 16 KB | 約 45 KB(React + ReactDOM) | Vue が約 **3分の1** |
| ルーター(必要時) | vue-router 約 11 KB | react-router-dom 約 15 KB | Vue が約 25% 軽 |
| 状態管理 | Pinia 約 6 KB | Zustand 約 3 KB / Redux Toolkit 約 14 KB | 同等〜React がやや軽 |
| 合計(典型構成) | **約 33 KB** | **約 63 KB** | Vue が約 30 KB 軽 |

初回ロードに 30 KB の差は、3G 回線(IT 音痴おばあちゃんの実環境)で **約 0.5
〜 1 秒の差** に相当する可能性。 PWA は初回ロード後にキャッシュされるが、
**「最初の体験」** の印象差は無視できない。

#### 1.2.2 ランタイム性能(JS Framework Benchmark 系)

| 項目 | Vue 3 | React 18 |
|---|---|---|
| 1000 行のリスト描画 | 速い(Proxy ベース細粒度更新) | 中程度(VDOM 全体差分) |
| 部分更新 | 速い(reactive システム) | 中程度(再 render + 差分) |
| メモリ使用量 | やや少なめ | やや多め |

チャット UI 程度では **両方とも十分高速** だが、Vue 3 は内部的な無駄が少ない。
スマホ低性能機種(古い Android、IT 音痴ユーザーのスマホ実態を想定) ではこの
差が「サクサク感」として体感に出る可能性。

#### 1.2.3 PWA + Capacitor 連携の成熟度

| 項目 | Vue 3 | React |
|---|---|---|
| Capacitor 公式サポート | ◎ 公式テンプレートあり | ◎ 公式テンプレートあり、事例最多 |
| Ionic 統合(必要時) | ◎ Ionic Vue 成熟 | ◎ Ionic React 成熟 |
| PWA プラグイン | ◎ vite-plugin-pwa(汎用) | ◎ vite-plugin-pwa(汎用) |
| 商品化事例 | ○ 多数あり | ◎ 最多 |

両方とも Capacitor 対応問題なし。 React の方が事例多いがVue でも実害なし。

### 1.3 Tさん テスト戦略 v0.1 §8.1 が React 想定で組まれている件

#### 1.3.1 影響評価

私(Sさん)は Tさん テスト戦略 v0.1 本文をまだ読んでいないため、 §8.1 が
React 前提でどの程度具体化されているかの精密評価は本書で完了できない。一般論
として:

- **E2E テスト(Playwright)**: フレームワーク非依存、影響なし
- **静的検査(NG/推奨ワード辞書、CI)**: フレームワーク非依存、影響なし
- **ユニットテスト**: Vue 3 は Vue Testing Library + Vitest、React は React
  Testing Library + Vitest。 **API は @testing-library/* 系で共通**、テスト
  ケース記述自体は移植コスト小
- **コンポーネント記述**: SFC(`.vue`)と JSX(`.tsx`)で違う、テスト戦略の
  「テスト対象例コード」が React のみで書かれている場合は Vue 版に書き直す
  必要あり

#### 1.3.2 対応案

- Tさん 側で v0.2 起草時に **「フロントエンド確定後にテスト戦略 v0.2 で具体例
  を更新」** の運用を提案
- 確定が遅れる場合、Tさん は「フレームワーク非依存部分」を v0.2 で先行確定、
  フレームワーク依存部分は v0.3 で更新
- 私が本書 §13(Q列)でフロントエンド確定をたかしさんに仰ぐので、確定後に
  Tさん 引き継ぎ時に明示

### 1.4 判断材料の整理(たかしさん向け)

| 観点 | Vue 3 優位 | React 優位 | Sさん 感触 |
|---|---|---|---|
| 動作の軽さ(バンドル、ランタイム) | ◎ 約30 KB 軽、Proxy ベースで速い | △ | **Vue 3 推奨理由** |
| PWA + Capacitor エコシステム | ○ 十分 | ◎ 事例最多 | 差は実装上問題にならない |
| 学習曲線 | ◎ SFC 構造で読みやすい | ○ JSX に慣れている人多い | Vue 3 推奨理由 |
| 求人市場・採用 | △ | ◎ | 本プロジェクトは個人開発、影響小 |
| Tさん テスト戦略の手戻り | △ React 前提部分の書き換え | ◎ そのまま使える | React にする理由になり得る |

#### 1.4.1 最終的なたかしさんの判断ポイント

「動作の軽さ重視」基準で考えると **Vue 3 推奨が崩れない** 。 React にするなら
「Tさん テスト戦略との手戻り回避」が決め手になるが、テスト戦略の手戻り工数は
Vue 3 採用で削減できる「初回ロード時間 + ランタイム性能差」のメリットを上回ら
ないと感触している(両方とも数日規模、本プロジェクトのスケジュール感では中性)。

### 1.5 推奨確定(v0.2)

- **第一推奨: Vue 3 + Vite + TypeScript + Pinia**(理由 §1.2 / §1.4)
- **副推奨: React + Vite + TypeScript + Zustand**(Tさん テスト戦略との
  整合性重視時のみ)
- ベータ v1.0 までの最小構成 / 商品化版の堅牢構成: いずれも同じフレームワーク
  を継続使用

## 5. データ保管 — Drive 完結方針(v0.2 全面書き換え)

### 5.1 v0.1 → v0.2 の方針転換

| 項目 | v0.1 | **v0.2** |
|---|---|---|
| ベータ v1.0 | ローカル(IndexedDB)中心 | **Drive を真実の源(SoT)、ローカルは性能キャッシュ** |
| Drive 同期 | Phase 3 後半に導入 | **Phase 2 第1スプリントから実装** |
| 移行作業 | ベータ → 商品化で必要 | **発生しない** |
| マルチデバイス | Phase 3 後半以降 | **ベータから対応** |

### 5.2 推奨アーキテクチャ — C ハイブリッド方式(仕様書 v1.5 §20 と擦り合わせ)

```
[Drive(SoT、真実の源)]                [ローカル(性能キャッシュ)]
 MIYU_App_Data/                         IndexedDB
 ├── config/                              ├── settings.json (キャッシュ)
 │   ├── settings.json (暗号化)            ├── consent.json (キャッシュ)
 │   ├── consent.json                     ├── characters/ (キャッシュ)
 │   └── characters/                      └── lastSyncedAt: タイムスタンプ
 │       ├── miyu.md(初期搭載)
 │       └── (商品化以降に追加)
 └── (memory/ は Phase 3 以降)
```

### 5.3 起動時の同期フロー

```
1. PWA 起動
2. ローカルキャッシュから設定を即座にロード(高速、オフライン対応)
3. オンラインなら Drive と比較
   - Drive の lastModified > ローカルの lastSyncedAt → Drive 版を取得、ローカル更新
   - ローカル変更がオフラインで発生していた → Drive にアップロード
   - 両方変更があった(マルチデバイス競合)→ 競合解決(§5.6)
4. ユーザー操作開始(設定変更等)
5. 変更は即座にローカル → 定期的(または重要操作後)に Drive 同期
```

### 5.4 オフライン対応

- 初回起動は **オンライン必須**(Drive から初期設定取得)
- 2回目以降は **オフラインでも起動可能**(ローカルキャッシュから)
- オフライン中の変更は **ローカルキャッシュにキュー**、オンライン復帰時に
  自動 Drive 同期
- 思想書「タッチ操作ができない人でも使える」哲学を担保: ユーザーは「同期した」
  「失敗した」を意識せず、UI 上は常に動く

### 5.5 Drive API レート制限への対応

- Drive API: 1秒あたり 10 リクエスト程度(個人利用は十分余裕)
- **デバウンス処理**: 設定変更が連続した場合、最後の変更から 2秒後にまとめて
  Drive 書き込み
- **バッチング**: 複数ファイル変更を1回の Drive API 呼び出しで実施

### 5.6 マルチデバイス競合解決

- ベータ v1.0 では **シンプルな Last-Write-Wins** で開始
- 家族テスター運用中に「マルチデバイス使用がそもそも発生するか」を観察
- 競合が頻発する場合、 Phase 4 以降に CRDT ベースなどの本格対策を検討
- 個人秘書アプリのユースケース上、 **同一ユーザーが2デバイスで同時編集する
  シナリオは稀** なので、 LWW で実用上問題ない見込み

### 5.7 BYOK API キーの暗号化(変更点)

| 項目 | v0.1 | **v0.2** |
|---|---|---|
| 保管先 | IndexedDB + WebCrypto AES-GCM | **Drive(暗号化)+ ローカルキャッシュ(暗号化)** |
| 鍵 | 端末固有 | **ユーザーパスフレーズ由来(復元可能)** |
| 復元 | 端末再インストールで失われる | **新デバイスでパスフレーズ入力で復元** |

ユーザーパスフレーズ由来の鍵を採用することで、 **マルチデバイス対応 + 端末
再インストール時の復元** が両立する。

### 5.8 退会フローとの整合(法的書類 v0.3 §6.10)

- ユーザーは **Drive の `MIYU_App_Data/` フォルダを手動削除** することで全
  データを消去可能
- PWA 側にも「退会(全データ削除)」ボタンを設置、 ボタンタップで Drive API
  経由でフォルダ削除を実行
- 法的書類 v0.3 §6.10「データ自己削除手順」 と整合

### 5.9 v0.1 で挙げたリスク(iOS Safari IndexedDB 7日ポリシー)の解消

- Drive を SoT にすることで、 IndexedDB がロストしても **Drive から再同期で
  復元** 可能
- 家族テスター運用前の「事前説明」も不要(ユーザーには見えない)
- 思想書「移行画面を見せない」哲学を担保

### 5.10 Phase 2 第1スプリント着手前に決めるべき細部

- Drive 内のフォルダ命名: `MIYU_App_Data/`(仕様書 v1.4 §5.1 通り) で確定?
- 暗号化アルゴリズム: WebCrypto AES-GCM-256(現代標準) で確定?
- Drive スコープ: `https://www.googleapis.com/auth/drive.file`(本アプリが
  作成したファイルのみアクセス可能、最小権限原則) で確定?

これらは仕様書 v1.5 §20 本文化と擦り合わせて確定する想定。

## 6. VOICEVOX 統合(v0.2 全面書き換え)

### 6.1 v0.1 → v0.2 の方針転換

| 項目 | v0.1 | **v0.2** |
|---|---|---|
| ベータ v1.0 | 見送り、 Web Speech API | **VOICEVOX 採用** |
| 商品化版 | クラウド VOICEVOX or 外部 TTS で再検討 | 同一アーキテクチャを継続、 §6.7 でキャラ追加 |

### 6.2 VOICEVOX 統合の選択肢(3案)

| 案 | 構成 | 評価軸: 立ち上がり | 評価軸: 運用負担 | 評価軸: 法的整合 |
|---|---|---|---|---|
| 1(推奨) | **VOICEVOX エンジンを Cloud Run に自前デプロイ、HTTPS API として PWA から呼ぶ** | ○ Docker 公式イメージあり、デプロイ半日 | ○ Cloud Run リクエスト時のみ起動、家族規模なら無料枠内 | △ 「専用サーバー」かは要解釈、本書 §13 Q で確認 |
| 2 | 第三者 VOICEVOX 互換クラウド API(VOICEVOX Cloud Plus 等、サービスがあれば) | ○ 即座に使える | ○ 運用負担なし | ○ 「第三者 AI サービス」枠で法的書類 §11 と整合 |
| 3 | 事前合成済み音声 + Web Speech フォールバック | △ 固定セリフのみ可、動的応答に弱い | ◎ サーバー不要 | ◎ 完全ローカル |

### 6.3 推奨: 案1 VOICEVOX エンジン自前 Cloud Run デプロイ

**根拠**:
1. **エンジン本体オープンソース、Docker イメージが公式提供**
2. **キャラクター全員の音声合成が可能**(クラウド API だとキャラ制限がある可能性)
3. **Cloud Run のリクエスト時起動方式** で、家族規模(月数千リクエスト程度)
   なら **無料枠内で運用可能** (Cloud Run の無料枠は月 200万リクエスト)
4. **遅延は許容範囲**: VOICEVOX 合成は概ね 0.5〜2秒、 Cloud Run の cold start
   は 1〜3秒。 **応答 1往復で合計 2〜5秒** 、家族規模では UX として許容
5. **第三者クラウド API は商用化条件が不安定** な可能性あり、自前の方が長期
   安定

### 6.4 ライセンス・クレジット表記

**ライセンス確認結果(エルトン書面で確認済み、 2026-05-18)**:
- VOICEVOX ソフトウェア自体: 商用・非商用問わず無料利用 OK
- **クレジット表記必須**(例: 「VOICEVOX:ずんだもん」「©VOICEVOX」)
- キャラクターごとに利用規約が異なる、 **営利利用に事前申請必要なキャラもあり**
- ベータ家族配布は「営利」でないので問題なし
- **商品化前に採用キャラ全員の個別利用規約をリーガルチェック必要**(別タスク、
  たかしさん実施)

### 6.5 アプリ内クレジット表記の表示位置(設計案)

- **設定画面 「情報」セクション**: 採用 VOICEVOX キャラ全員の権利表記を一覧表示
- **キャラクター選択画面**(§6A 参照): 各キャラ選択肢の右下に「VOICEVOX:キャラ名」を
  小さく表示
- **AI 明示宣言 起動時 UI**(法的書類 §6.1)とは別画面で OK、 AI 明示宣言は
  「AI による応答」の宣言、 VOICEVOX クレジットは「音声合成エンジンの権利表記」
  なので役割が違う

### 6.6 採用キャラ個別利用規約確認の手順(別タスク、たかしさん実施)

たかしさんが各キャラを採用する前に実施する手順:
1. VOICEVOX 公式ホームページの「キャラクター」ページから採用候補キャラの
   利用規約 URL を取得
2. 各キャラの利用規約を確認(配布アプリ内利用 / クレジット表記 / 営利利用 /
   その他の特殊条件)

### 6A.3 キャラクター差し替えのデータ構造

各キャラのメタデータ:

```typescript
interface Character {
  id: string;                    // 例: 'miyu' / 'miwa-nesan'
  displayName: string;           // 例: 'MIYU' / '美輪ねえさん'
  characterMdPath: string;       // Drive 内パス、例: 'config/characters/miyu.md'
  voicevoxSpeakerId: number;     // 例: 1(ずんだもん:あまあま)
  voicevoxCreditLine: string;    // 例: 'VOICEVOX:ずんだもん'
  description: string;           // 選択画面で表示する短い紹介
  bundledInBeta: boolean;        // ベータ v1.0 初期搭載かどうか
}
```

### 6A.4 character.md の動的読み込み(Drive 経由)

- Drive 内 `MIYU_App_Data/config/characters/<id>.md` に各キャラ定義を配置
- ベータ初回起動時、 PWA が **初期2キャラの character.md を Drive に書き込む**
  (PWA バンドルに埋め込まれた初期コンテンツから)
- ユーザーがキャラを切り替えた瞬間、 character.md 内容を読み込み、
  systemInstruction を再構築
- ユーザーが Drive 上で character.md を編集すれば、 PWA が次回起動時にその
  変更を反映(パワーユーザー機能、ベータでは積極的にアピールしない)

### 6A.5 キャラクター選択UIの設計

- 起動時の設定画面に「キャラクター選択」ボタン
- タップすると **キャラ一覧画面**(2キャラのカード形式、サムネ + 名前 +
  ボイスサンプル「こんにちは〜」)
- 各キャラの右下に **VOICEVOX クレジット表記**(小さく)
- 「決定」を押すと選択保存、初期挨拶画面に遷移
- **3択ボタン式確認**(仕様書 v1.4 §22) は不要(キャラ選択は破壊的操作で
  ないため)
- **音声品質を実感できる**「ボイスサンプル」 を画面に置くことで、ユーザーが
  「このキャラが好き」 と感じる体験を提供(IT 音痴ユーザーへの UX 配慮)

### 6A.6 キャラ追加の拡張性(Phase 4 以降)

- 商品化フェーズで追加キャラ販売(仕様書 v1.2 §9.2 案2)時、 各キャラの
  character.md と VOICEVOX 話者 ID マッピング、メタデータJSON を追加するだけ
- アプリコア機能の改修は不要(キャラは設定駆動)
