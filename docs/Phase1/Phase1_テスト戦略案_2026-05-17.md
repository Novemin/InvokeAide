## 1. テスト戦略の原則

### 1.1 5つの基本原則

1. **法的必須要件は仕様/実装/テストの三位一体で担保する** 。法的書類で約束したことは、テストで検証可能な形に落とす。
2. **スマホ音痴対応 UX は人間の感性を経由したテストが必須** 。自動化だけでは届かない。
3. **「未来の自分を縛らない」をテスト表現にも適用** 。テスト名・コメントでも断定を避け、「現時点では〜を確認」と書く。
4. **MoSCoW 法でスコープを絞る** 。ベータ v1.0 までは Must / Should、Could/Won't は明示的に後送り。
5. **テスタブルな実装を Sさん に依頼可能な形で示す** 。テスト不可能な実装は品質が測れない、両者協調が必要。

### 1.2 テスト4分類 + 法的書類要件テスト

指示書 §7.2 の4分類に法的要件テストを別建てで追加:

| 分類 | 役割 | 主な自動化 |
|---|---|---|
| **機能テスト** | 個別 API・関数の正常系/異常系 | ユニットテスト |
| **統合テスト** | 複数モジュール・外部 API 連携 | 統合テスト、モック |
| **セキュリティテスト** | 認証情報、通信、入力サニタイズ | 静的解析 + 動的検証 |
| **UX テスト** | 操作性、アクセシビリティ、感性 | 半自動(Lighthouse) + 手動 |
| **法的書類要件テスト** | 規約画面の表現、データ送信先、退会フロー | 静的検査 + ネットワーク監視 |

## 4. 自動化可否判定

### 4.1 判定区分

| 区分 | 定義 | 想定ツール |
|---|---|---|
| 🟢 **完全自動化可** | コード単体・ロジック・静的解析・API モックで完結 | Vitest / Jest、Playwright、ESLint、axe-core |
| 🟡 **ハイブリッド** | 自動化と手動を組み合わせる(自動: 機能、手動: 感性・実機・実 API) | Playwright + 実機テスト、Lighthouse + 視覚レビュー |

## 6. スマホ音痴対応 UX テスト方法(別建て)

ベータ v1.0 で「妥協しない」項目。家族テスター(スマホ音痴を含む)を想定し、定量・定性の両方で検証する設計。

### 6.1 ペルソナ(5名想定)

| ペルソナ | 想定属性 | 主な制約 |
|---|---|---|
| **P1. おばあちゃん(70代女性)** | IT音痴、初めてのアプリ操作、文字小さいと見えない | タッチも音声も慣れていない、複雑な遷移は理解しない |
| **P2. おじいちゃん(60代男性、視力低下)** | スマホは持っているが頻繁には使わない | 大画面表示・聞き取り音量が必要、文字色のコントラスト感受性 |

## 7. 法的書類要件テスト方法(別建て)

法的書類 v0.3 で約束した内容を、テストで検証可能な形に落とす。「言ったこと」と「実装」と「テスト」の三位一体を担保。

### 7.1 H1 / H7: 「未来の自分を縛らない」原則の静的検査

#### 7.1.1 仕組み

利用規約 HTML / マークダウン文書を入力として、 **NG ワード辞書** と **推奨ワード辞書** を持つ静的解析ツールを CI で実行。

```
入力: docs/規約_利用規約等統合版_*.md, public/legal/*.html
NG 例: "存在しない", "一切〜しない", "アクセスできない", "100%", "永久に〜しない", "絶対に"
推奨例: "運用していない", "通常運用において", "手段を持たない", "合理的な", "現時点では"
出力: 違反箇所(行番号 + 該当文 + 推奨置換)
```

#### 7.1.2 実装案

- 軽量実装: Node.js + 正規表現の単独スクリプト(`scripts/check-legal-expressions.js`)
- CI 統合: PR ごとに走らせる、違反があれば PR にコメント
- 辞書のメンテナンス: 法的書類 v0.3 §1.2 / §8.9 の置換ルール表をソース・オブ・トゥルースに

#### 7.1.3 注意点

- **誤検知への配慮**: 「存在しない概念」のような技術的記述で NG ワードが含まれることがある。 **法的書類セクション内のみ検査** とスコープ限定するか、 **NG ワードの前後文脈チェック** を入れる
- **多言語対応**: 英語版作成(C案フェーズ)時は辞書を別途整備

### 7.2 H2: 運営者ドメインへの POST 監視

#### 7.2.1 仕組み

Playwright で E2E テスト中の **全ネットワークリクエスト** を記録し、運営者ドメイン(将来 `*.novemintelligence.com` 等)への POST が **テスト終了時に 0件** であることを assert。

#### 7.2.2 実装案

```javascript
// 擬似コード
const operatorRequests = [];
page.on('request', (req) => {
  if (req.url().includes('novemintelligence.com') && req.method() === 'POST') {
    operatorRequests.push(req);
  }
});
// シナリオ実行(全機能を一通り叩く)
await runFullScenario(page);
// アサーション
expect(operatorRequests).toHaveLength(0);
```

#### 7.2.3 許可リスト機構(将来拡張)

法的書類 v0.3 §6.5「将来追加される可能性のあるデータ収集」が明示的に余地を残しているので、許可リストを **環境変数または config で更新可能** にする。例:

```
ALLOWED_OPERATOR_ENDPOINTS=crash-reports.novemintelligence.com,license.novemintelligence.com
```

許可されたエンドポイントへの送信は許可、それ以外は引き続き 0件 assert。

### 7.3 H3 / A1: AI 明示宣言の起動時表示

#### 7.3.1 仕組み

E2E テストで初回起動シーケンスを実行、最初の画面に **AI 明示宣言テキストが含まれる** ことを検証。

```javascript
// 擬似コード
await page.goto('/');
const aiDisclosure = page.locator('text=/本アプリは AI を用いた対話システムです/i');
await expect(aiDisclosure).toBeVisible();
```

#### 7.3.2 注意点

- 言語/文言の確定が前提。法的書類 v0.3 §6.1 の **「本アプリは AI を用いた対話システムです」** が暫定文言。エルトンの本文起草で確定後に固定。
- **2回目以降の起動** で AI 明示宣言を出すかは設計判断。法的書類は「起動時 UI に組み込む」と要求しているので、初回のみで足りるか、毎回出すかは別検討(本書 §10 判断仰ぎ事項に転記)。

## 11. 副次的な気づき

### 11.1 思想書 §4「IT 音痴対応」はテスト戦略の第一原則

思想書 §4「タッチ操作ができない人でも、声で注文できる」は、本テスト戦略 §6 UX テスト方法の **設計上の魂** 。ペルソナ P1, P2 の存在意義そのものであり、UX テスト成功基準(S1 完走率 100%)もここから逆算しています。第22章 入力UX 本文化時にも、思想書 §4 を「テスト基準の根拠」として組み込めると整合性が高い。

### 11.2 Phase 0 デモ UX 観察記録の不足

引継ぎ帳 §1「夕方(銀座食事会デモ)」で「お店の騒音で詳しい説明はできず → 学びとして記録」とあるが、 **デモ実施時の友人の反応の細部** は引継ぎ帳・仕様書に不足。第22章 入力UX 本文化時、もしくは家族テスト時に再収集すると貴重なインプット。

### 11.3 テスト用 Google アカウントは将来のサーバー側運営にも使える資産

法的書類 v0.3 §6.5「将来追加される可能性のあるデータ収集(クラッシュレポート、利用統計、ライセンス認証)」を実装する時、 **Novem Intelligence 名義のテスト用 Google アカウント** が運営者側通信先のテストにも使えます。今投資して取得する価値は中長期的にあります。

### 11.4 ベータ v1.0 配布は「家族との対話のきっかけ」としての価値もある

たかしさんの「7月家族集まりに間に合わせたい」というモチベーションは、単に **動くものを見せる** 以上に、 **家族との会話のテーマとしてアプリを使う** 価値があります。アプリそのものが家族時間を豊かにする一回目の演出 — テスト戦略上の成功は「バグがない」だけでなく「家族の対話を妨げない」も含むと感じます。

### 11.5 v1.0 → v1.x 自動アップデート前提は「妥協OK」を増やす

「PWA 自動アップデートで段階的に進化」前提なので、 **ベータ v1.0 では完璧を目指さず、家族が楽しめる最低限** を目指す方針は健全。Should 14項目は本当に余力あれば、Must 52項目を 100% 確実に動かす方が優先度高い。
