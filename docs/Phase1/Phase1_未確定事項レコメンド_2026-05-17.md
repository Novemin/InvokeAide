## 1. 論点1 — アプリ名「秘書召喚アプリ(仮称)」で問題ないか

### 1.1 結論

**開発内通称としては Yes、商品化前に思想書整合の再検討を推奨。**

### 1.2 技術的観点(私のスコープ内で)

呼称はコード上は変数名・コメント・UI 文言・OAuth 同意画面表記・利用規約・
ストア掲載名 など複数箇所に波及する。 **開発初期に1度確定** すれば良いが、
**商品化フェーズに入ってから変更** すると差し替え漏れ・履歴ノイズが多発する。
よって「商品化前」が改名の最終締切。

### 1.3 私が気付いた違和感(技術担当の越境ですが報告)

仕様書 §0.2 と思想書「父から息子へ」、および character.md(MIYU の人格定義)
を読み比べると、 **「秘書」という呼称が示す関係性** と **コアキャラクター設計
で謳う関係性** にニュアンス差がある:

| | 「秘書」の含意 | コアキャラクター設計の含意 |
|---|---|---|
| 役割 | 仕える側、サポート役 | もう一人の自分、対等な親友、並走者 |
| 関係性 | ヒエラルキーあり(主-従) | フラット(対等) |
| 言動 | 「承知しました」「お任せください」 | 「〜じゃん」「やば」「サクッと拾い、サクッと返す」 |
| 由来 | 業務支援文化 | Urban Hippies「愛・自由・和」 |

「秘書召喚」は **言葉の響きの面白さ・ファンタジー感** で確かに印象的だが、
「ユーザーが召喚するのは秘書」と言うとき、ユーザーは「上に立つ側」になる。
これは「対等な親友」という宣言と微かに矛盾する。

### 1.4 たかしさんの判断のための材料

- 開発内通称として継続するならコード・UI 等に影響なし
- 商品化前の正式名検討時の候補(私が考えた範囲、参考):
  - **同じ目線系**: 「(もう)もう一人」「相棒」「右腕」「Companion」
  - **召喚の遊び心系**: 「召喚」「Invoke」(コードベース名 InvokeAide 由来、思想書「ドリンク1杯で AI 召喚」とも整合)
  - **存在感系**: 「いる」「在(あ)る」 など、和語の名詞化
- 「秘書」が良いと感じる場合の理由を言語化しておくと、後の判断が早い。
  たとえば「ストア検索ヒット率」「初見ユーザーへの分かりやすさ」「『召喚』
  という遊びとの組み合わせ」 などの観点

## 2. 論点2 — Calendar 便利機能(attendees, colorId, conferenceData, attachments)を MVP 除外で良いか

### 2.1 結論

**4機能すべて Yes(MVP 除外)、ただし優先度に強弱あり。**

### 2.2 個別評価

| 機能 | MVP 除外 | 再検討タイミング | 根拠 |
|---|---|---|---|
| attendees(参加者招待) | ◎ Yes、強い | Phase 4 以降または常時保留 | 個人秘書としては不要、「人物データベース」設計が必要で MVP には重い |
| colorId(色分け) | ◎ Yes | Phase 3 で「タグ機能」として再検討 | 視覚整理用、UI フレームワーク選定後の検討が筋 |
| conferenceData(Meet) | ○ Yes | Phase 4 で需要観察後判断 | リモートワーク派には便利だが、ターゲット層(IT 中程度、おじいちゃんおばあちゃん含む)では優先度低 |
| attachments(Drive 添付) | ○ Yes | Phase 3 の Drive 統合フェーズで自然に検討 | Drive 連携と一体設計が綺麗、MVP で独立実装するメリットなし |

### 2.3 提案 — 「除外したものを再考するタイミング」をフェーズ計画に組み込み

仕様書では「Phase 3 以降検討」「Phase 4 以降検討」と書かれているが、 **どの
Phase の何のレビューで再検討するか** が明示されていない。 **MVP 除外しっ放し
で気付いたら誰も触らない** という事態を避けるため、Phase 計画に「除外機能の
再評価ポイント」を組み込んでおくと安全。

例:
- Phase 3 中盤レビュー: colorId / attachments の再評価
- Phase 4 ベータ配布後: attendees / conferenceData の需要観察

### 2.4 「友人からの要望」フィードバック回路

たかしさんが論点に挙げた「友人からの要望次第で前倒し」を実現するには、
**継続的なフィードバック収集の仕組み** が必要。Phase 4 ベータ配布前に:
- Polling フォーム or Feature Request 経路の整備
- 「この機能が欲しい」と気軽に投げられる UI(MIYU 経由で「次の機能要望ある?」
  と聞く UX も技術的には可能)

## 3. 論点3 — Tasks サブタスク(階層化)は商品化時の差別化要素になるか

### 3.1 結論

**条件付き Yes。機能単独では差別化弱い、「MIYU が階層を提案する」とセットで
初めて差別化価値あり。**

### 3.2 機能単独だと差別化弱い理由

- Google Tasks 自体が `parent` パラメータでサブタスク標準対応
- Google Tasks 公式アプリ・スマホ標準 Todo アプリも階層UIを既に持つ
- 「サブタスク機能を持つ」だけでは差別化要素にならない

### 3.3 「MIYU が階層を提案する」なら強い差別化

**AI による構造化提案** は Google Tasks 標準 UI では実現できない領域。

具体例:

```
ユーザー: 「引越し準備、ToDo に入れて」
MIYU(差別化なし): 「了解、『引越し準備』追加したよ〜」
MIYU(差別化あり): 「了解、『引越し準備』追加した〜。これ、ザッと
                  細かく分けると『段ボール調達』『電気手続き』『住所変更』
                  『不用品処分』とかかなって思うんだけど、サブタスクで
                  ぶら下げとく?」
```

これは:
- Google Tasks 単体では出来ない
- 既存の To-Do アプリには無い体験
- 思想書「軽めのつまみある? と聞ける」「自然言語で AI に聞ける」
  という哲学とも整合
- character.md「能動的に問いを返す」とも整合

### 3.4 実装コスト見積もり

- **parent パラメータ追加**: 低(Function Declaration 1引数追加)
- **AI 階層提案ロジック**: 中(systemInstruction にプロンプト追記 + 提案結果
  を3択ボタンで確認する UX)
- **階層UI**: 中(リスト表示で1段インデント、UI フレームワーク選定後実装)

3つを合わせると、Phase 3 後半の1〜2スプリント規模。

### 3.5 たかしさんの判断のための材料

- 「MIYU が階層を提案する」UX に魅力を感じますか? それとも過剰機能ですか?
- 提案頻度: 毎回提案するか、ユーザーが望んだ時だけか(後者の方が UX 負担少)
- 3択ボタンで「①ぶら下げて / ②独立で / ③じゃあ自分で考える」のような確認
  UI を経由するなら、第22章 入力UX と統合設計可

## 4. 論点4 — 第26章 配信ロードマップの各 Phase 切り替えタイミング

### 4.1 結論

**路線は適切、3点の微調整を提案。**

### 4.2 現行ロードマップの再掲

| Phase | 主な配信方法 | 補助 |
|---|---|---|
| Phase 0(完了) | ② ngrok 一時公開 | ① PC ローカル |
| Phase 1〜3 | ② ngrok or ③ Cloudflare Pages | 機能拡張優先 |
| ベータ配布 | ④ PWA | ③ も併用検討 |
| 正規版商品化 | ④ PWA + ⑤ React Native 検討 | App Store 申請も視野 |

### 4.3 微調整提案1 — Phase 1〜3 で ngrok 継続は基本不要

- 開発時は **Vite dev server などのローカル HMR 環境** で十分(ホットリロード、
  デバッグ環境統一)
- ngrok は **「他端末でのプレビューが必要な時だけ起動するスポット使用」** に
  限定した方がよい
- 理由: ngrok 常設はセキュリティ運用負担(Basic 認証管理、URL ローテーション、
  認証情報ローテーション)が継続発生する、開発スピードに不利

### 4.4 微調整提案2 — Cloudflare Pages の導入を Phase 2 後半に早める

- 設定画面・PWA インストール体験・Service Worker キャッシュ挙動は **本番に近い
  環境(HTTPS、HTTP/2、Service Worker 有効)** で早期検証したい
- ngrok 環境だと URL が毎回変わって設定保存テストが煩雑、PWA のインストール
  体験も挙動が異なる
- Cloudflare Pages 自体は数分でデプロイ可能、無料、永続 URL なので導入コスト
  低

### 4.5 微調整提案3 — Capacitor 採用判断は Phase 4 中盤に後ろ倒し

- PWA で出てくる **iOS 制約**(プッシュ通知の制限、Web Speech API の挙動差、
  バックグラウンド処理の不可) を **観察してから判断** する方が安全
- 先に Capacitor を決め打ちすると「Capacitor で解決される問題が、実は PWA
  だけで解決できた」というオーバーキルになり得る
- Capacitor 化のコストは「PWA を完成させた後、薄いラッパーを被せる」だけ
  なので、 **後ろ倒ししても遅くない**

### 4.6 修正後ロードマップ(私の提案)

| Phase | 主な配信方法 | 補助 |
|---|---|---|
| Phase 0(完了) | ② ngrok 一時公開 | ① PC ローカル |
| Phase 1 | Vite dev server(ローカル HMR) | 必要時のみ ngrok 起動 |
| Phase 2 前半 | Vite dev server | |
| **Phase 2 後半** | **③ Cloudflare Pages 導入** | Vite dev server も継続 |
| Phase 3 | ③ Cloudflare Pages + 設定画面/PWA UX 検証 | |
| ベータ配布 | ④ PWA(Cloudflare Pages 配信) | |
| **Phase 4 中盤** | **PWA 運用観察 → ⑤ Capacitor 採用判断** | |
| 正規版商品化 | ④ PWA + ⑤ Capacitor(判断次第) | App Store 申請 |

### 4.7 たかしさんの判断のための材料

- 修正後ロードマップに異論ありますか?
- 「Vite dev server」は Step 3「技術スタック決定提案」で別途確定する論点なので、
  本書ではあくまで「ローカル HMR 環境の例」として書いています

## 6. 論点6 — v1.5 で本文化する章の優先順位(第20〜23章)

### 6.1 結論

| 章 | 優先度 | v1.5 本文化推奨 |
|---|---|---|
| 第22章 入力UX設計思想 | 🔴 最優先 | ◎ Yes |
| 第20章 設定値保存 | 🔴 最優先 | ◎ Yes |
| 第21章 マニュアル運用 | 🟡 中 | △ v1.5 か v1.6 |
| 第23章 ブランド構想 | 🟢 低 | ❌ 商品化前まで凍結可 |

### 6.2 第22章 入力UX設計思想 を最優先する理由

- **3択ボタン式確認(①お願いします / ②やらない / ③保留)** は、Phase 2
  第1スプリント(設定画面 + AI 明示 + 年齢確認 + 規約同意) で **同意ボタン
  系の UI コンポーネント設計と直結**
- 法的書類 §6.4 / §8.4 の「重要操作は確認 UI を経由する」要件の **実装担保**
- 削除機能を解禁する前提(デモは 3択UI 未実装のため削除非対応)
- 入力方法の選択肢(音声・タップ・キーボード) は OS 標準音声入力依存設計に
  関わるので、設計確定が早いほど Phase 2 着手がスムーズ

### 6.3 第20章 設定値保存 を最優先する理由

- BYOK モデルの **API キー保管設計** が決まらないと、Phase 2 第1スプリントの
  設定画面が組めない
- 「Cハイブリッド方式(Google Drive を真実の源 + ローカルキャッシュ)」と
  v1.3 構成案で言及されているが、 **本文化が必要**:
  - Drive 同期の頻度(リアルタイム / 起動時 / 手動)
  - 競合解決(マルチデバイスで設定変更が衝突した場合)
  - 暗号化方式(キーは Drive 上でもローカルキャッシュ上でも暗号化必須)
  - オフライン時の挙動

### 6.4 第21章 マニュアル運用 を中優先とする理由

- ユーザー向けマニュアル(`config/manual.md`)は Phase 4 ベータ配布前に整備
  すれば間に合う
- 開発と文書化を一体化する運用方針自体は良いが、 **本文化のタイミングは
  Phase 2 着手の必須条件ではない**
- v1.5 か v1.6 のどちらでもよい。「v1.5 で第20章・第22章を本文化する負荷が
  重い場合は v1.6 に回す」という運用が現実的

### 6.5 第23章 ブランド構想 を低優先とする理由

- ClubFreedomTokyo サブドメイン、Novem Intelligence、9尾の白狐ロゴ等は
  **商品化フェーズの話**
- 開発フェーズ中盤(Phase 2〜3)までは凍結可能
- ロゴ等の検討は **仕様書本体ではなく別ファイル** で並行進行が良い
- 仕様書に取り込むのは商品化前(Phase 4 後半)で十分

### 6.6 たかしさんの判断のための材料
