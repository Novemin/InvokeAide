## 4. 5/26 スコープ議論で必要な観点(議題提案、 判断は議論者に委ねる)

5/26 議論を建設的にするため、 以下の観点を整理しておくことを提案します。 各観点に対する判断は 5/26 議論で行ってください。

### 4.1 観点 1 — 「6/20」 の解釈

- (a) **絶対期日**: 6/20 を超えない、 スコープ削減を必須とする(シナリオ X / Y / Z のいずれかを選択)
- (b) **目標期日(数日オーバー許容)**: 6/20 を目標とし、 数日オーバー(6/23 程度)は許容する(シナリオ 0 維持 + ダメ出し期間圧縮を受容)
- (c) **絶対期日 + ダメ出し期間延長**: 7月頭家族配布のスケジュールを動かしてダメ出し期間を確保する(本筋ではないが選択肢として)

### 4.2 観点 2 — ダメ出し期間 2週間の性質

- (a) **機能検証重視**: 全機能(音声 + キャラ別 + コーチング)を見たいので、 シナリオ 0 もしくは シナリオ Y 寄り
- (b) **UX 検証重視**: 機能の厚みより UX の流れを見たいので、 シナリオ X もしくは シナリオ Z 寄り
- (c) **両方**: 機能・UX 両方を見たい(理想的だが期間内に収まるか別途検討)

### 4.3 観点 3 — 信頼度ターゲット

- (a) 信頼度 8割以上を取りたい → シナリオ Z(余裕大)
- (b) 信頼度 7割程度で良い → シナリオ X
- (c) 信頼度 5-6割を受容する → シナリオ 0(現状維持)

### 4.4 観点 4 — ベータの位置づけ

- (a) **「秘書召喚アプリ」 のコンセプト全体を体感させる版** → 音声 + キャラ別 を維持(シナリオ 0 / シナリオ X)
- (b) **「コーチング機能」 を中心に体感させる版** → 音声・キャラ別は v1.1 でも可(シナリオ Y / シナリオ Z)
- (c) **機能網羅性を最大化、 ベータ後の差分を最小化** → シナリオ 0
- (d) **完成度を上げて配布、 ベータ後の補強を許容** → シナリオ X / Y / Z

### 4.5 観点 5 — 削減項目の優先順位

v0.4 §8.5 では「VOICEVOX > セバスチャン の順で削減」 を Sさん 推奨としたが、 これは 6/30 基準での感触。 6/20 基準では削減量がより重要になるため、 5/26 議論で改めて優先順位を確認することを提案。

選択肢:
- (a) VOICEVOX > セバスチャン(v0.4 §8.5 継承)
- (b) セバスチャン > VOICEVOX(キャラ差別化を v1.1、 音声は維持)
- (c) 両方同時に後送り
- (d) 両方維持(シナリオ 0)

## 5. 副次的気づき(本メモのスコープ外、 議論材料として)

### 5.1 「ダメ出し期間」 自体のスコープ定義

「約2週間ダメ出し」 の中で具体的に何を検証するかが明示されていません。 たかしさん側で:
- 機能テスト(全機能が動くこと)
- UX テスト(家族テスター層に通用するか)
- 法的要件確認(年齢確認 / AI 明示 / Privacy Policy 表示)
- セキュリティ確認(BYOK 鍵管理 / Drive 完結の挙動)

のうちどれを主軸とするか、 5/26 議論で確認することを提案。 これによりベータの最終的な「合格基準」 が明確になります。

### 5.2 シナリオ X 採用時の「Web Speech API フォールバック」 の品質基準

シナリオ X(VOICEVOX 後送り)を採用する場合、 Web Speech API による音声合成フォールバックがベータ品質として十分か別途検証が必要。 Web Speech API は OS / ブラウザ依存で品質ばらつき大、 iOS Safari と Android Chrome で読み上げ品質が大きく異なる可能性。 落とし穴集 v0.2 で実機検証(Sprint 3 想定)を待つか、 シナリオ X 採用時は事前検証を Sprint 2 に組み込むか、 議論余地あり。

### 5.3 シナリオ Y 採用時の「キャラ別口調検証の代替」

シナリオ Y(セバスチャン 後送り)を採用する場合、 「キャラ別口調こそ差別化の肝」(v0.4 §6.3)の検証がベータでできなくなる。 代替案:
- たかしさん 個人テスト時にセバスチャン 開発版を試して内部検証
- ベータ家族テスター向けに「セバスチャン 近日対応」 として期待感を醸成
- v1.1 リリース時に「キャラ追加」 を目玉機能化

これらは 5/26 で議論せず、 v1.1 計画段階の論点として保留可。

### 5.4 ダメ出し期間が長くなる(シナリオ Z)場合の運用

シナリオ Z(両方後送り、 ダメ出し期間 約20日)を採用する場合、 ダメ出し期間が長くなることで:
- たかしさん の負担増(検証時間)
- Sさん・Uさん の「ダメ出し → 修正」 サイクルの稼働期間延長
- 修正で発見された大きな問題への対応余地が広がる

これは「長いダメ出し期間 = 安全マージン大」 とも見える一方、 「修正の連鎖でベータ品質が動的に変わる」 リスクもあるため、 5/26 議論で運用ルール(修正の打ち切り日設定等)を確認することを提案。

### 5.5 「6/20」 という日付の象徴的意味
