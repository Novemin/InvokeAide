以下を CLAUDE.md 末尾(現 §10 「このファイル自体の運用」の前)に **新規 §10 として挿入** 。挿入により既存 §10 は §11 に番号繰り下げ(変更履歴セクションは末尾のまま)。

```markdown
## 10. 命名規則とドキュメント参照規約

本プロジェクトのドキュメント命名と相互参照の規約。詳細・選択肢評価・移行案は
`docs/Phase1/Phase1_命名規則統一案_2026-05-17.md` を参照。

### 10.1 ファイル命名テンプレート

```
<プロダクト名 | 分類>_<サブ用途>_<v?.?>[_<ドラフトマーカー>]_<YYYY-MM-DD>.md
```

- プロダクト名: `秘書召喚アプリ`(本プロダクト)、`Gemini版MIYU`(旧称、変更しない)等
- 分類: `思想書` / `引継ぎ帳` 等、プロダクト名ではないがそれに準ずる
- サブ用途: `仕様書` / `法的書類起草指示書` / `開発計画指示書` 等
- バージョン: **ピリオド表記** (`v1.4`、`v0.3`)
- ドラフトマーカー: 必要時のみ `_ドラフト`(日本語マーカー、案A)
- 日付: ISO 8601(`2026-05-16`、ハイフン)

### 10.2 章番号参照規約(版間衝突の回避)

- 仕様書 / 法的書類 / 指示書を参照する時は **版番号併記必須**
  - 例: `仕様書 v1.4 §20`、`法的書類 v0.3 §6.5`
- 思想書 / CLAUDE.md / inception_context.md は版番号がないか改訂頻度が低いため簡略可
  - 例: `思想書 §4`、`CLAUDE.md §1.4`
- 理由: 仕様書 v1.2 §20 = 変更履歴、v1.4 §20 = 設定値保存 のように、版間で章番号が再配置されるため

### 10.3 新規ファイル作成時のチェックリスト

- [ ] §10.1 テンプレートに従っているか
- [ ] バージョン表記がピリオド(`v1.4`)になっているか
- [ ] 日付が ISO 8601(`2026-05-17`)か
- [ ] プロダクト名と用途の間に区切り `_` があるか
- [ ] ドラフト状態なら `_ドラフト` マーカーがあるか

### 10.4 段階的移行(v0.2 以降)

`STさん指示書_秘書召喚アプリ開発計画_v0_1_2026-05-17.md` 等、現状アンダースコア表記のファイルは v0.1 を現状維持。**次バージョン改訂時に §10.1 テンプレート(プロダクト名先頭 + ピリオド表記)へ移行する** 。
```
