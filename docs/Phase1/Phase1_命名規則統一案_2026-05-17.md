## 2. 命名規則の選択肢評価

### 2.1 バージョン表記

| 案 | 例 | 採用率(実ファイル) | 評価 |
|---|---|---|---|
| 案A: ピリオド | `v1.4` | 3/5(60%) | SemVer 風で標準的 / ソート時の振る舞いが直感的 / Markdown レンダリングで違和感なし |
| 案B: アンダースコア | `v1_4` | 2/5(40%) | URL safe / 一部ファイルシステムで安全性高 / SemVer 慣習と乖離 |
| 案C: ハイフン | `v1-4` | 0/5 | DNS や URL でよく見る / 「v1-4」は範囲指定っぽく見える誤読リスク |

**推奨**: **案A(ピリオド)** 。理由は SemVer 標準準拠・ファイル数の少数派が現状アンダースコアで移行コスト小・Markdown ドキュメント内表記との一貫性。

### 2.2 プレフィックス・命名パターン

| 案 | 例 | 採用率(実ファイル) | 評価 |
|---|---|---|---|
| 案A: 用途プレフィックス型 | `仕様書_秘書召喚アプリ_v1.4_2026-05-16.md` | 0/7 | ソート時に種類別にまとまる(全仕様書が並ぶ) |
| 案B: プロダクト名プレフィックス型 | `秘書召喚アプリ_仕様書_v1.4_2026-05-16.md` | 2/7 | ソート時にプロダクトごとにまとまる |
| 案C: 用途とプロダクトを連結(現状の主流) | `秘書召喚アプリ仕様書_v1.4_2026-05-16.md` | 2/7 | 簡潔だが、用途とプロダクトの境界が不明瞭 |
| 案D: ファイル横断的に揺れあり(現状)| (現状の混在状態) | 7/7 | 解消対象 |

**推奨**: **案B(プロダクト名プレフィックス型)** 。理由はマルチプロダクト体制(現行 MIYU / InvokeAide / 呑呑呑 / 法的書類整備)で **プロダクトごとにファイルがまとまる方が4プロジェクト並行体制と整合** 。

### 2.3 ドラフト/確定マーカー

| 案 | 例 | 評価 |
|---|---|---|
| 案A: 日本語マーカー | `_ドラフト` / `_確定` | 日本語ドキュメントとの整合 / 視覚的に明示 |
| 案B: 英語マーカー | `_draft` / `_final` | 英語版作成時の継承可 / 短い |
| 案C: 短縮 | `_dr` / `_fn` | 短い / 可読性低 |
| 案D: マーカーなし(版番号で状態管理) | (なし)| シンプル / 版0.x = ドラフト、v1.0+ = 確定 等の暗黙ルール頼り |

**推奨**: **案A(日本語マーカー、必要な時のみ付与)** 。理由は日本語ドキュメント前提のプロジェクトで視覚的に明示しやすい。版番号で暗黙にする(案D)よりミスが少ない。

### 2.4 区切り文字

| 案 | 例 | 評価 |
|---|---|---|
| 案A: アンダースコア統一 | `秘書召喚アプリ_仕様書_v1.4_2026-05-16` | 統一性◎ / 日付がアンダースコアだとISO 8601 から外れる |
| 案B: ハイフン統一 | `秘書召喚アプリ-仕様書-v1.4-2026-05-16` | 統一性◎ / URL safe / 日本語と混ぜると見にくい |
| 案C: ミックス(プレフィックス=アンダースコア、日付=ハイフン)| `秘書召喚アプリ_仕様書_v1.4_2026-05-16` | 現状の主流、日付は ISO 8601 標準 |

**推奨**: **案C(ミックス)** 、現状を踏襲。

## 3. 推奨命名規則(統合案)

### 3.1 命名テンプレート

```
<プロダクト名 | 分類>_<サブ用途>_<v?.?>[_<ドラフトマーカー>]_<YYYY-MM-DD>.md
```

要素:
- プロダクト名: `秘書召喚アプリ`(本プロダクト) / `Gemini版MIYU`(旧称、変更しない)等

- サブ用途: `仕様書` / `法的書類起草指示書` / `開発計画指示書` 等
- バージョン: `v1.4`(ピリオド統一)
- ドラフトマーカー: `_ドラフト`(任意、必要時のみ)
- 日付: `2026-05-16`(ISO 8601、ハイフン)

## 6. 文書内章番号参照規約

### 6.1 問題: 版間で章番号が衝突

| 章 | 仕様書 v1.2 | 仕様書 v1.4 |
|---|---|---|
| 第20章 | 変更履歴 | 設定値保存 |
| 第27章 | (なし) | 変更履歴 |

「仕様書 §20」と書くと、版によって意味が違う罠が発生。

### 6.2 規約案

#### 6.2.1 必須ルール

- **仕様書を参照する時は版番号併記必須**: 例 `仕様書 v1.4 §20`
- **法的書類を参照する時は版番号併記必須**: 例 `法的書類 v0.3 §6.5`
- **他文書(指示書、引継ぎ帳)も同様、版番号併記推奨**

#### 6.2.2 簡略可ルール

- **思想書はバージョン上書きの可能性が低いので簡略可**: `思想書 §4`
- **CLAUDE.md / inception_context.md は固定名で版番号がないので、章記号のみで OK**: 例 `CLAUDE.md §1.4`

### 6.3 既存文書での順守状況

Tさん の Phase 1 成果物(整合チェック・テスト戦略・本レポート)では、本規約をすでに自然に適用済み(`仕様書 v1.4 §20`、`法的書類 v0.3 §6.5` 等)。今後の文書でも踏襲する。

## 7. 将来のファイル作成時の運用ルール

### 7.1 命名チェックリスト

新規ファイル作成時、以下を確認:

- [ ] ファイル名が §3 命名テンプレート `<プロダクト名>_<サブ用途>_<v?.?>[_<ドラフトマーカー>]_<YYYY-MM-DD>.md` に従っているか
- [ ] バージョン表記がピリオド(`v1.4`)であるか、アンダースコア(`v1_4`)になっていないか
- [ ] 日付が ISO 8601(`2026-05-17`)であるか
- [ ] プロダクト名と用途の間に区切り `_` があるか
- [ ] ドラフト状態なら `_ドラフト` マーカーがあるか

### 7.2 文書作成時の参照チェックリスト

文書内で他文書を参照する時:

- [ ] ファイル名参照は実ファイル名と完全一致しているか
- [ ] 章番号参照は版番号併記しているか(`仕様書 v1.4 §20`)
- [ ] CLAUDE.md / inception_context.md / 関連文書での該当ファイルへの参照とも整合しているか

### 7.3 リネーム実施時のプロセス

1. リネーム実行(`Rename-Item` / `git mv` 等)
2. 関連文書の参照表記更新(grep で全件検出してから一括修正)
3. インデックス文書(`CLAUDE.md §4`、`memory/MEMORY.md` 等)の参照更新
4. 引継ぎ帳に変更履歴を残す(将来の参照のため)
