### 0.2 提案の通底原則

1. **ベータまでは「最小構成で完走」を優先**: 動くものを早く作り、家族テスター
   から学ぶ
2. **抽象化レイヤーだけは最初から入れる**: 後付けが効きにくい層( AI 抽象化、
   設定保管、TTS 切替)は初期設計で interface 化
3. **見送り判断を恐れない**: VOICEVOX / Apps Script / Drive 完結 を「商品化
   版で再検討」と明示することで、ベータ完成を優先
4. **法的書類 v0.3 §1.2「未来縛らない原則」を技術選定にも適用**: 「永遠に X」
   を選ばず、「現時点で X、状況変化で再検討」を明示

## 1. フロントエンドフレームワーク

### 1.1 候補3案

| 案 | フレームワーク | 評価軸: 立ち上がり速度 | 評価軸: モバイル UI | 評価軸: TypeScript 親和性 | 評価軸: Capacitor 対応 |
|---|---|---|---|---|---|
| 1 | Vue 3 + Vite | ◎ Composition API + SFC で構造化しやすい、HMR 高速 | ◎ 標準で十分、UI ライブラリ豊富(Vant, Quasar 等) | ○ 公式 TS 対応、`<script setup lang="ts">` で簡潔 | ○ Capacitor 公式が Vue を例示、Ionic Vue 経由でも可 |
| 2 | React + Vite | ◎ エコシステム最大、サンプル豊富 | ◎ UI ライブラリ最も豊富(MUI, Chakra, Tamagui 等) | ◎ TS 親和性は最高、型推論が成熟 | ◎ Capacitor 事例が最多、Ionic React 経由でも可 |
| 3 | SolidJS + Vite | ◎ React 風だが軽量、Signals で reactive | ○ UI ライブラリは React より少ない | ○ TS 対応良好 | △ Capacitor 事例が少なく、トラブル時の調査時間が長い |

### 1.2 推奨: 案1 Vue 3 + Vite

**根拠**:
1. **6週間で動くものを作る** という制約下で、 SFC の構造化された記述は新規
   開発者でも追いやすい(Phase 2 第1スプリントが複雑、設定画面 + AI 明示 +
   年齢確認 + 規約同意 + チャット UI が並走)
2. **Composition API + Pinia(状態管理)** の組み合わせがチャット系 UI と
   相性良好(履歴管理、ストリーミング応答、function calling ループ)
3. **学習コストの低さ** が、本プロジェクトの「DAO 的フラット組織」と相性
   良い(Tさん や将来加わるメンバーも触りやすい)
4. **Capacitor 対応も十分**(Ionic Vue 経由か直接連携、商品化版で iOS / Android
   ラッパー化する道は確保)

### 1.3 副推奨: 案2 React + Vite

**Vue ではなく React を選ぶ場合の判断材料**:
- Sさん や開発チームに **React 経験が圧倒的に多い** 場合
- 商品化版で React Native への移行も視野に入れる場合(現状は Capacitor 路線
  で確定なので、この優位性は希薄)
- 求人市場・学習リソースの広さを最重視する場合

技術的優劣ではなく **チーム familiarity** で決まる論点なので、たかしさんの感触
で覆して問題ない。

### 1.4 案3 SolidJS は不採用理由

- パフォーマンス優位はあるが、ベータ v1.0 でその差を活かす場面が少ない
- Capacitor 事例が少なく、商品化版で詰まった時の調査コストがリスク
- スピード重視のフェーズでエコシステム小さい選択は不利

### 1.5 ベータ v1.0 までの最小構成 / 商品化版の堅牢構成

| 区分 | 構成 |
|---|---|
| ベータ v1.0 | Vue 3 + Vite + Pinia + vue-router(必要時) |
| 商品化版 | 同上 + Capacitor ラッパー(iOS / Android)+ UI ライブラリ統合(必要時) |

ベータと商品化で **同じフロントエンドコアを使う** ので、立ち上がりから商品化
まで再構築不要。

## 2. TypeScript 採用有無

### 2.1 推奨: 採用

**根拠**:
1. **AI 抽象化レイヤーの interface 設計** が型で表現できる(後述 §4)。これが
   今回最も大きい。
2. **法的書類要件のテスト** で型ガードが効く。「年齢確認フラグ」「規約同意
   フラグ」のような重要な状態を `boolean` で散らさず型で守れる
3. **スマホ音痴向け UX** で「予期せぬ undefined」を防ぐ第一防衛線。家族
   テスター運用前の品質ゲート(memory: feedback-ux-three-layer 層1)と整合
4. Vue 3 / React どちらも TS 親和性が高く、 **学習コストは導入コストとして
   一度払えば後は楽になる** (ベータ 6週間の中でも回収可能)

### 2.2 デメリットと対処

| デメリット | 対処 |
|---|---|
| 初期セットアップに半日程度かかる | Vite テンプレート(`pnpm create vite@latest -- --template vue-ts`)で 5分 |
| 型定義に時間取られる | 厳しすぎる設定は避ける(`strict: false` から始めて段階的に上げる)|
| ライブラリの型定義が無い場合がある | DefinitelyTyped(`@types/*`)で大半カバー、無い場合は ambient 宣言で済ます |

### 2.3 設定方針(技術的詳細、エルトン参考用)

- `tsconfig.json`: `"strict": true`、 `"noUncheckedIndexedAccess": true`
- ベータでは any 一時使用も許容(コメント `// TODO: tighten in v1.x`)
- 型と実装の整合は CI で `tsc --noEmit` を回す(memory: feedback-test-三位一体
  の「静的検査」相当)

## 3. バックエンド構成

### 3.1 候補3案

| 案 | 構成 | 評価軸: 立ち上がり | 評価軸: ベータ後の進化 | 評価軸: 運用負担 |
|---|---|---|---|---|
| 1 | フロントエンド完結(バックエンドなし)、必要に応じて Apps Script を Phase 3 以降 | ◎ 即着手可、サーバー要らない | ◎ Apps Script で朝サマリ等の非同期処理を後から追加可能 | ◎ 運用負担ほぼゼロ |
| 2 | 最初から Apps Script ベース | △ Apps Script の制約(6分実行)を考慮した設計が必要 | ○ サーバーレス哲学と整合 | ○ Google アカウント内完結、運用負担小 |
| 3 | Node.js + Express(Phase 0 server.js 系譜) | ○ Phase 0 経験あり、開発速度速い | △ 結局 Apps Script に移行コストが発生する見込み | ❌ ホスティング必要、運用負担大 |

### 3.2 推奨: 案1 フロントエンド完結で開始 + Apps Script を Phase 3 以降に必要分のみ

**根拠**:
1. **Phase 0 デモが実証済み**: バックエンドゼロでも Gemini API 直叩き、
   Calendar/Tasks(OAuth refresh_token)直叩きで動く
2. **BYOK モデルと整合**: API キーは端末内保管、運営者サーバー不要、法的書類
   v0.3 §6.5「専用サーバーを運用していない」表現と整合
3. **ベータ v1.0 で「動くもの」を最短で作るため**: 6週間でサーバー設計・
   ホスティング・運用までやる余裕は薄い
4. **将来の Apps Script 導入余地は残る**: 朝サマリ・階層メモリ圧縮など「定時
   実行が必要な処理」は Phase 3 以降に Apps Script で段階導入

### 3.3 ベータで実装する API 呼び出しは全てフロントエンドから

| 機能 | 呼び出し先 | 認証 |
|---|---|---|
| LLM(Gemini 等) | Gemini API(generativelanguage.googleapis.com)直叩き | ユーザー BYOK API キー |
| Calendar 操作 | Calendar API 直叩き | OAuth(ユーザー本人) |
| Tasks 操作 | Tasks API 直叩き | OAuth(ユーザー本人) |
| Drive 同期(Phase 3 以降) | Drive API 直叩き | OAuth(ユーザー本人) |

### 3.4 商品化版で追加検討

- **クラッシュレポート受信**: Cloudflare Workers / Pages Functions で軽量に。
  ベータでは不要(法的書類 v0.3 §6.5 で「将来導入する場合は規約改訂と事前
  通知」を明示済み)
- **使用統計**: 同上、商品化フェーズで必要に応じて
- **Apps Script**: 朝サマリ生成、階層メモリ daily → weekly 等の圧縮バッチ

### 3.5 たかしさん追加方針(Q4)との整合

「形になったら走らせる」を実現するため、 **Apps Script 設計に時間をかけずに
フロントエンド完結で走り始める** のが、本論点で最も効く判断。

## 4. AI 抽象化レイヤー interface 設計案

### 4.1 設計目的(再確認)

法的書類 v0.3 §11「運営者は技術的・商業的判断に基づき、他のAIサービス(Anthropic
Claude、OpenAI GPT、xAI Grok、ローカル LLM 等)に変更または併用する可能性が
ある」 を **実装で担保** する。

### 4.2 候補3案

| 案 | パターン | 評価 |
|---|---|---|
| 1 | アダプタパターン(各 LLM プロバイダを共通 interface に適合させる)+ ベータは Gemini 1実装のみ | ◎ シンプル、後から実装追加可、過剰設計回避 |
| 2 | ファクトリ + ストラテジー + アダプタの組み合わせ | △ ベータ段階では過剰、機能追加で利益が出る Phase 4 以降に検討 |
| 3 | 直接呼び出し、抽象化なし | ❌ 法的書類 §11 と非整合、却下 |

### 4.3 推奨: 案1 アダプタパターン、ベータは Gemini 1実装のみ

### 4.4 interface 設計草案(TypeScript)

```typescript
// === Core types ===
export interface LLMProvider {
  readonly name: string;
  readonly model: string;
  generateContent(params: GenerateParams): Promise<GenerateResponse>;
  supports(feature: LLMFeature): boolean;
}

export type LLMFeature =
  | 'functionCalling'
  | 'grounding'
  | 'streaming'
  | 'systemInstruction';

export interface GenerateParams {
  systemInstruction: string;
  messages: Message[];
  tools?: ToolDefinition[];
}

export interface Message {
  role: 'user' | 'model' | 'tool';
  parts: MessagePart[];
}

export type MessagePart =
  | { type: 'text'; text: string }
  | { type: 'functionCall'; name: string; args: unknown }
  | { type: 'functionResponse'; name: string; response: unknown };

export interface ToolDefinition {
  name: string;
  description: string;
  parameters: JSONSchema;
}

export interface GenerateResponse {
  text: string;
  functionCalls?: FunctionCall[];
  groundingSources?: GroundingSource[];
  rawResponse?: unknown; // デバッグ用、ユーザー側に露出させない
}

// === Implementations ===
export class GeminiProvider implements LLMProvider {
  readonly name = 'gemini';
  readonly model: string;
  constructor(opts: { apiKey: string; model: string }) { /* ... */ }
  async generateContent(params: GenerateParams): Promise<GenerateResponse> { /* ... */ }
  supports(feature: LLMFeature): boolean { /* ... */ }
}

// Phase 4 以降に追加
// export class ClaudeProvider implements LLMProvider { ... }
// export class OpenAIProvider implements LLMProvider { ... }

// === Factory(設定駆動)===
export function createProvider(config: ProviderConfig): LLMProvider {
  switch (config.kind) {
    case 'gemini': return new GeminiProvider(config);
    // case 'claude': return new ClaudeProvider(config);
    // case 'openai': return new OpenAIProvider(config);
    default: throw new Error(`Unknown provider: ${(config as { kind: string }).kind}`);
  }
}
```

### 4.5 設計上のポイント

1. **`Message.parts` は配列型**: Gemini の `parts` 形式と整合、Claude の
   `content` 配列とも変換可能。OpenAI の単一 `content` 文字列は変換時に
   `[{ type: 'text', ... }]` でラップ
2. **`functionCalls` を `GenerateResponse` に含む**: function calling
   ループ(Phase 0 デモ:1018-1136 行 = MIYU_demo.html の sendToGemini 関数)
   と互換性を保つ
3. **`supports(feature)` で機能差を吸収**: 例えば Claude の grounding は
   別アーキテクチャ(MCP等)なので、 `supports('grounding') === false` と
   返してアプリ側でフォールバック判定
4. **`rawResponse` はデバッグ用、ユーザー側には露出しない**: 法的書類 §6.5
   「通常運用において収集・保存しない」と整合、本番ビルドでは undefined

### 4.6 ベータ v1.0 までの最小構成 / 商品化版

| 区分 | 実装 |
|---|---|
| ベータ v1.0 | interface + `GeminiProvider` 1実装 |
| Phase 4 以降 | `ClaudeProvider` / `OpenAIProvider` を必要時に追加 |
| 商品化版 | 設定画面で「使う AI を選択」UI、API キーは各プロバイダごとに別保存 |

### 4.7 Phase 0 デモからの知見継承

Phase 0 デモ(`MIYU_demo.html:1036-1137`)の `sendToGemini` 関数は実質的に
`GeminiProvider.generateContent` に相当する処理を含む。本 interface に
リファクタする際の参考実装として開いたまま運用する。

## 5. データ保管

### 5.1 仕様書 v1.4 §4 / §5 / §20(構成案)との整合性

仕様書: Google Drive 内完結、設定3ファイル + 階層メモリ4フォルダ、 C ハイブリッド

### 5.3 推奨: 案1 段階導入

**根拠**:
1. **ベータ v1.0 で Drive 連携設計までやる余裕は無い**: Drive 同期は競合解決・
   オフライン挙動・暗号化 など考慮事項が多く、6週間では設計負債を抱えるリスク
2. **iOS Safari の IndexedDB 7日ポリシー** (落とし穴集 §4.6)を「データ

3. **Phase 2 第1スプリント案(設定画面 + AI 明示 + 年齢確認 + 規約同意 +
   モック1往復) はローカルストレージで完結する**

### 5.4 ベータ v1.0 でのデータ層構成

| 種類 | 保管先 | 暗号化 |
|---|---|---|
| API キー(BYOK) | IndexedDB(または localStorage)+ WebCrypto AES-GCM 暗号化、鍵は端末固有(ユーザーパスフレーズ由来)| ◎ 必須 |
| OAuth トークン(refresh_token 含む) | 同上 | ◎ 必須 |
| ユーザー設定(年齢確認済み・規約同意済みフラグ等)| localStorage | △ 不要(機密でない)|
| 会話履歴 | in-memory(Phase 0 デモと同じ、リロードで消える)| ◎ 不要(保存しない)|
| 同意ログ | IndexedDB(法的書類対応で要監査記録)| △ 不要 |

### 5.5 Phase 3 後半で導入する Drive 同期

- 設定値の Drive 同期(JSON ファイルとしてユーザー Drive 内 `MIYU_App_Data/
  config/` に保存)
- 仕様書 §20「C ハイブリッド方式」の本実装
- マルチデバイス対応はこの時点で初めて可能になる

### 5.6 商品化版で追加検討

- 階層メモリ(daily/weekly/monthly/yearly)の Drive 配置
- Apps Script による圧縮バッチ(daily → weekly)
- character.md / profile.md / dictionary.md の Drive 配置と動的読込

## 6. VOICEVOX 統合方法

### 6.1 候補3案

| 案 | ベータ v1.0 | 商品化版 |
|---|---|---|
| 1(推奨) | 見送り、Web Speech API | VOICEVOX クラウド or 外部 TTS(ElevenLabs 等)で再検討 |
| 2 | ベータから VOICEVOX(サーバー必要、現実的でない) | 同上 |
| 3 | ベータから外部 TTS(ElevenLabs)で「ぽい声」 | VOICEVOX に置換 or 継続 |

### 6.2 推奨: 案1 ベータでは Web Speech API、商品化版で再検討

**根拠**:
1. **VOICEVOX 公式 API はローカル動作前提**、スマホアプリから直接叩く構造に
   なっていない傾向
2. **クラウド VOICEVOX サービス** は出始め時期で商用利用条件が安定していない
   可能性、リスクとして抱えにくい
3. **運営者サーバーで VOICEVOX をホストする方式** は §3.2 のサーバーレス志向
   と非整合、運用負担大
4. **引継ぎ帳「新論点3」** で「音声品質投資(VOICEVOX 改善 / Live API /
   ElevenLabs)」と並列に挙げられている通り、 **VOICEVOX 固定は決まっていない**
5. **ベータ家族テスター期間に「Web Speech の棒読み問題がどれだけ嫌われるか」
   を観察してから投資判断する方が筋が良い** (memory: feedback-ux-three-layer
   層3「家族テスターによる感性レビュー」と整合)

### 6.3 設定画面の構造を「TTS プロバイダ切替可能」にしておく

interface レベルで TTS も抽象化:

```typescript
export interface TTSProvider {
  readonly name: string;
  speak(text: string, options?: SpeakOptions): Promise<void>;
  cancel(): void;
  isAvailable(): Promise<boolean>;
}

export class WebSpeechTTSProvider implements TTSProvider { /* ... */ }
```

## 7. STT 実装

### 7.1 iOS Safari の制約再確認(落とし穴集 §4.4)

- **Web Speech Recognition API は iOS Safari で非対応の傾向**
- iOS 標準の音声入力(キーボード上のマイクボタン)は textarea で使える

### 7.2 候補3案

| 案 | ベータ v1.0 | 商品化版 |
|---|---|---|
| 1(推奨) | OS 標準音声入力依存(textarea にマイクで入力 → ユーザーが ↩ 押す) | 同上 + 必要に応じて外部 STT(Whisper 等)を追加検討 |
| 2 | Web Speech Recognition API + Whisper API ハイブリッド | 同上 |
| 3 | ローカル Whisper(WebAssembly) | 同上 |

### 7.3 推奨: 案1 OS 標準音声入力依存

**根拠**:
1. **仕様書 §22(本文化待ち、v1.5)** で「スマホ OS 標準の音声入力機能を活用」
   と既に方向性確定
2. **追加実装コスト ゼロ**: textarea を置けば iOS / Android 両方で OS 標準
   音声入力が使える
3. **法的書類 §6.5「通常運用において収集・保存しない」と整合**: 外部 STT
   サービスに音声データを送らない構造
4. **思想書「タッチ操作ができない人でも声で注文できる」哲学とも整合**: OS
   標準なので、ユーザーは普段使い慣れた音声入力方法を継続できる

### 7.4 商品化版で追加検討

- 文中にマイクボタンを置く UX に進化させる場合は Whisper API 等の外部 STT
- 音声入力の精度評価(memory: feedback-ux-three-layer 層2 半自動・観察記録)
  の結果次第
