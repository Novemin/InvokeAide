## 1. 通知設計(新章)

### 1.1 要件整理

たかしさん指示書 §2.3:
- **18:00**(仕事終わり想定、 設定で調整可推奨)に コーチング通知
- 発火条件: Google Tasks に未完了タスクがある日
- 通知媒体: Telegram(既存 PC1 MIYU 基盤) または 別手段
- 通知 → ユーザーがアプリを開く → MIYU/セバスチャンと一緒にタスク見直し

### 1.2 候補3案

| 案 | 通知媒体 | バックエンド依存 | iOS/Android 対応 | ベータ v1.0 適合 |
|---|---|---|---|---|
| 1(推奨) | **Google Calendar Event リマインダー** | なし(FE完結 + Calendar API) | ◎ OS ネイティブ通知 | ◎ |
| 2 | Telegram Bot(既存 PC1 MIYU 基盤を流用) | Telegram API + Apps Script 等 | ◎(Telegram アプリ必要) | △ 家族にスマホ音痴がいる、 Telegram 導入ハードル |
| 3 | Web Push API(PWA + Service Worker) | 自前 Push サーバー必要 | iOS 16.4+ PWA インストール時のみ | △ iOS 制約、 バックエンド必要 |

### 1.3 推奨: 案1 Google Calendar Event リマインダー

#### 1.3.1 根拠

1. **FE 完結を維持**(v0.2 §3 バックエンド方針との整合)、 自前 Push サーバー不要
2. **仕様書 v1.4 §7 と整合** — そもそも通知インフラとして Calendar Event を選択していた経緯がある
3. **OS ネイティブ通知** — iOS / Android 標準の通知 UI、 IT 音痴ユーザーにも馴染みやすい
4. **Telegram 不要** — 家族テスター(memory: feedback-ux-three-layer)に Telegram 導入を強いずに済む
5. **MIYU 専用カレンダー** (仕様書 §7.3 自動作成)を活用可、 ユーザーのメインカレンダーを汚染しない

#### 1.3.2 設計フロー

```
PWA 起動時(または1日1回バッチ):
  1. MIYU 専用カレンダー(初回起動時に作成済み)に
     「MIYU - 18:00 タスク見直し」 イベントを翌日分(または今日 18:00 前)作成
  2. リマインダー: { method: 'popup', minutes: 0 } で 18:00 ジャストに通知
  3. イベント本文に「📱 アプリを開いて MIYU と整理しよう」 リンク
  4. 既存の同イベントがあれば更新、 重複作成しない

18:00 ユーザーのスマホで通知発火:
  1. OS の Calendar 通知が表示
  2. ユーザーがタップ → ブラウザ or PWA 起動
  3. PWA が「コーチングモード」 で起動
     - Google Tasks から未完了タスクを取得
     - ランク別に集計(A:○件、 B:○件、 C:○件、 D:○件)
     - 選択中キャラの口調でプッシュメッセージ生成
  4. MIYU / セバスチャン がチャット冒頭で発言
```

#### 1.3.3 設定可能項目

- **通知時刻**: 18:00 デフォルト、 設定画面で調整可(時刻スライダー)
- **通知頻度**: 毎日 / 平日のみ / カスタム曜日
- **通知 OFF**: 設定で完全に切れる
- **通知条件**: 「未完了タスクがある日のみ」(デフォルト) / 「毎日(タスクなしでも声かけ)」

### 1.4 副案 — Telegram Bot 連携(将来オプション)

ベータ v1.0 後の v1.1 で、 たかしさん本人(パワーユーザー)向けに Telegram Bot 連携をオプション追加可能。 PC1 既存 MIYU の Telegram Bot 基盤と統合する道筋。 ベータ v1.0 では不要。

### 1.5 Web Push API は本筋ではない

- iOS 16.4+ + PWA インストール状態でしか動かない、 家族の環境制約大
- 自前 Push サーバー必要、 v0.2 §3 FE完結方針と矛盾
- ベータ v1.0 では除外、 商品化版で再検討

## 2. タスク管理コーチング機能(新章)

### 2.1 要件整理(指示書 §2.2 / §2.3 / §3.1)

- ランク別(A/B/C/D)の促し方
- 「これCランクだけど、まだやる予定ある?」(削除提案)
- 「Aの〇〇、いつやる?」(実施日決定促進、 ただし時刻指定不要)
- 「すぐできないならBに下げる?」(ランク変更促進)
- 「重要じゃないならC?」(ランク変更促進)
- **キャラ別口調で発動**(MIYU 砕け / セバスチャン フォーマル)

### 2.2 プッシュロジック設計

#### 2.2.1 起動時のタスク状態分析

PWA がコーチングモードで起動すると、 サービス層が以下を計算:

```typescript
interface TaskCoachingContext {
  // ランク別集計
  countByRank: { A: number; B: number; C: number; D: number };
  // ランク × 経過日数
  staleTasks: {
    A: Task[];  // 3日以上経過した A
    B: Task[];  // 7日以上経過した B
    C: Task[];  // 7日以上経過した C(削除候補)
    D: Task[];  // 7日以上経過した D(削除候補)
  };
  // 期限切れタスク
  overdue: Task[];
  // 今日完了したタスク
  completedToday: Task[];
}
```

#### 2.2.2 プッシュ優先順位ロジック

以下の順序で、 1回のコーチング内で最大3〜4論点を取り上げる(情報過多回避):

| 優先 | トリガー | プッシュ内容 |
|---|---|---|
| 1 | 期限切れタスクあり | 「期限過ぎてるやつあるよ、 どうする?」 |
| 2 | A ランクで 3日以上経過 | 「Aの〇〇、 いつやる?」 |
| 3 | B ランクで 7日以上経過 | 「Bの〇〇、 A に上げる? C に下げる?」 |
| 4 | C/D ランクで 7日以上経過 が3件以上 | 「C/D で溜まってるやつ、 整理しよ?」 |
| 5 | 今日完了タスクあり | 「今日 〇〇 終わったね、 お疲れ!」(肯定的締め) |

優先 1 〜 4 を順に取り上げ、 最後に 5 で肯定的に締める(MIYU/セバスチャン キャラ共通の「ジャッジしない」 軸との整合)。

#### 2.2.3 ユーザー回答の受け取り

各プッシュ項目に対して、 ユーザー回答は 3択ボタン式UI(仕様書 §22 / v1.5 §22 本文化と整合)で受け取る:

| プッシュ | 3択ボタン |
|---|---|
| 「いつやる?」(A 滞留) | ①今日 / ②明日 / ③今週中 / ④もう少し考える |
| 「ランク変えるか?」(B 滞留) | ①A に上げる / ②C に下げる / ③そのまま |
| 「整理する?」(C/D 滞留) | ①消す / ②残す / ③1件ずつ確認 |
| 「期限過ぎてるが」 | ①完了にする / ②期限延ばす / ③消す |

#### 2.3.3 プッシュテンプレートを character.md とは別ファイル化する提案

**理由**:
- character.md は **性格定義の本質**(価値観、 振る舞いの軸、 禁則)
- プッシュテンプレートは **文脈駆動の動的生成**(変数差し込み、 ランク別分岐)
- 両者を同じファイルに混ぜると character.md が肥大化、 編集時の認知負荷が上がる

**提案ファイル構成**:

```
MIYU_App_Data/config/characters/
├── miyu.md                       (性格定義)
├── miyu.coaching.md              (コーチングテンプレート)
├── sebastian.md                  (性格定義)
├── sebastian.coaching.md         (コーチングテンプレート)
└── index.json                    (メタデータ)
```

systemInstruction 構築時に、 性格定義 + コーチングテンプレートを連結する:

```typescript
function buildCoachingPrompt(character: Character, context: TaskCoachingContext): string {
  const characterMd = loadCharacterMd(character.id);          // 性格定義
  const coachingMd = loadCoachingMd(character.id);            // コーチングテンプレート
  const interpolated = interpolate(coachingMd, context);      // {title} 等を埋める
  return `${characterMd}\n\n---\n\n${interpolated}`;
}
```

### 2.4 「いつやる?」 への回答 と 時刻指定の扱い(私の所感、 §5 で詳述)

たかしさん指示書「時刻指定までは不要で『期限』 の管理だけでいい」 と、 プッシュロジック「Aの〇〇、 いつやる?」 のあいだに **設計上の小さなテンション** があります。 ユーザーが「明日 19時にやる」 と答えた場合の処理は §5 で議論します。

## 5. Sさん 視点での所感(率直、 指示書 §4 で歓迎)

たかしさん指示書 §4 で「Sさん の率直なフィードバックを求めています」 と明示して
くださっているので、 遠慮なく書きます。

### 5.1 設計上の小さなテンション — 「期限のみ」 vs 「いつやる?」

たかしさん指示書 §2.3 で「時間指定までは不要で『期限』 の管理だけでいい」 と
明示されています。 一方、 プッシュロジック §2.2.2 では「Aの〇〇、 いつやる?」 と
問う必要がある。

ここで「いつやる?」 への回答パターン:
- (a) ユーザー「今日やる」 → Tasks の due を今日に更新、 単純
- (b) ユーザー「明日やる」 → due を明日に更新、 単純
- (c) ユーザー「明日 19時にやる」 → ここで時刻情報が出てくる、 どう扱う?

**Sさん 提案**:
- ベータ v1.0 では (c) のような時刻入り回答に対し、 **「日付だけ取って、 時刻は
  メモに入れる」**(MIYU 例: 「19時とのことね、 メモに入れとくね!」)
- Tasks API の time 制約(write OK / read NG)とユーザー体験のずれは、 メモ
  フィールドへの退避で吸収
- 商品化版で「時刻もちゃんと扱う」 形に進化させる際は、 Calendar Event 化
  (仕様書 §17.4「機が熟したら昇格」)の手動オプションを追加

これで「時刻指定不要、 期限のみ」 たかしさん指示を尊重しつつ、 ユーザーの
時刻表現も拾える。 過剰設計にはならない。

### 5.2 PC1 既存 MIYU との同期は単方向(InvokeAide → MIYU)を推奨

§3.3 で触れた通り、 character.md / coaching.md のマスター管理は InvokeAide 側
に置く方が綺麗。 理由:
- PC1 既存 MIYU はたかしさん個人用、 InvokeAide は商品化版、 設計の中心は
  InvokeAide 側
- character.md を Drive に置くと、 PC1 既存 MIYU が Drive 経由で読み取り
  可能(エンタープライズアカウントなら Apps Script 経由でも可)
- 編集競合を回避するため、 PC1 側で character.md を書き換える必要があれば、
  InvokeAide 側に編集申請を出す運用

ただし、 これは Aさん/Bさん の運用にも影響するので、 エルトン経由で擦り合わせ
が必要。

### 5.3 ランク自動付与(仕様書 §17.4)はベータでは「提案」 まで

仕様書 §17.4「新規 ToDo のランク設定は MIYU が自動付与」 と書かれていますが、
ベータ v1.0 では **「MIYU/セバスチャンが『これ C ランクっぽいけどどう?』 と
提案する」 まで** が現実的:
- 自動付与の精度を上げるには学習データ(過去のランク付与履歴) が必要、 ベータ
  では蓄積不足
- ユーザー上書きを必須にすると煩雑、 提案 + 3択ボタン受諾の UX が好ましい
- 商品化版で学習機能を導入する余地を残す

### 5.4 プッシュ頻度の過剰化リスク

18:00 毎日通知 + 毎回 4論点プッシュ = ユーザー側の「通知疲れ」 リスクあり。
家族テスター(memory: feedback-ux-three-layer 層3) でこれを観察、 1週間で
「通知 OFF にしたい」 が出る可能性。 対策案:
- **プッシュ密度のスマート化**: 「同じタスクを 3日連続でプッシュしない」 等
  のクールダウン
- **完了肯定の比重を上げる**: 「お疲れ〜」 の割合を増やして、 ジャッジ感を
  薄める
- **「今週はプッシュ控えめモード」 設定** を追加(週単位で通知頻度調整)

これらはベータ v1.0 では実装せず、 観察結果次第で v1.1 で追加検討。

### 5.5 エルトン §2.1 経緯への共感

指示書 §2.1 で「エルトンが情報統合不足で見落としていた」 という正直な共有
ありがとうございます。 コーチング機能は確かに「秘書召喚アプリ」 のコアコン
セプトで、 v0.1 / v0.2 で見落としていたのは、 私もエルトンも同じ盲点を踏んで
いた可能性。 私が v0.2 §6A を書く時点で、 「秘書 = プッシュする存在」 という
本質を意識していれば、 もっと早く問題提起できたかもしれません。

これは feedback メモリ `feedback_publicity_and_concept_check.md` で記録した
「コアコンセプト整合チェック」 の運用テストでもありました。 命名だけでなく、
**機能設計の段階でも同じチェックを通す** よう、 自分の運用を強化します。

### 5.6 私からの異論はなし

エルトンの方針 §2.1〜§2.5 に異論はありません。 すべて方向性として合意。 上記
§5.1〜§5.5 は実装時の細部・運用上の懸念であって、 大方針への異論ではないこと
を明示。
