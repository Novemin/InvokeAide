### 1.2 🟢 低優先: §4.3 `@@CHOICES ... @@END` マーカーの代替案2案併記

**現状**: Sさん §4.3 で `@@CHOICES ... @@END` 構造化マーカーを LLM 出力規約として明示済み。

**提案**: Uさん UI コンポーネント実装時に選択しやすくするため、 **代替フォーマット2案を §4.3 末尾に併記** :

| 案 | フォーマット例 | 評価 |
|---|---|---|
| 現状(Sさん 案)| `@@CHOICES \n - value: "..." label: "..." \n @@END` | YAML 風、 LLM の理解度高 |
| 代替案A: YAML 形式 | ``` ```yaml \n choices: \n   - value: today \n     label: 今日 \n ``` ``` | YAML 標準、 パース容易 |
| 代替案B: Markdown リスト + メタデータ | `**Choices**: [today=今日, tomorrow=明日]` | コンパクト、 LLM 直感的 |

Uさん UI コンポーネント実装時に最も実装しやすいフォーマットを選ぶ余地が広がります。本提案は **Sさん vs Uさん の擦り合わせを促進する** だけで、 Sさん 案がベストである可能性も高いです。

**工数目安**: 30分(調査 + 併記)

### 1.4 🟡 中優先: §3.4 「温度のある演技 OK」vs 「実在人物 NG」境界例の充実

**現状**: Sさん §3.4 で `✅ 「マジで?それはちょっと大変だね」` / `❌ 「私もそういう経験あるよ」` 等、 2例で境界線を示している。

**提案**: 境界例を **5〜10個に拡充** することで、LLM の判断材料が厚くなり、出力の予測可能性が上がります。Tさん 観点で「LLM 任せ方針(§2.2)で安定性を高める」効果あり。

**Tさん 想定追加例(任意)**:

| パターン | ✅ OK | ❌ NG |
|---|---|---|
| 状況への反応 | 「マジで?それはちょっと大変だね」 | 「私もそういう経験あるよ」 |
| 機微への踏み込み | 「失礼ながら、少し気になります」 | 「私も心配しております」 |
| 共感表現 | 「うまくいかない時もあるよね」 | 「あたしも同じ気持ち」 |
| 完了肯定 | 「お疲れ、よく頑張ったね」 | 「私もうれしい」 |
| 提案 | 「整理してみる?」 | 「私がやっておきたい」 |

**Tさん 想定追加文**:

> キャラ演技は「状況への自然な反応」 まで OK 、 「キャラ自身の人生経験・感情・意思」 の主張は NG 。 例えば「マジで?」 のような相槌は MIYU の口調として OK だが、 「私もそういう経験あるよ」 は擬人化誤認。 「失礼ながら気になります」 は機微への言及として OK だが、 「私も心配しております」 は感情主張で NG 。

**工数目安**: 30分
