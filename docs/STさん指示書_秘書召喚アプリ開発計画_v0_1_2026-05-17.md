## 1. プロジェクト全体像と方針アップデート

### 1.1 たかしさんのプロジェクト体制(冒頭、重要)

Sさん・Tさんに前提として共有したい、たかしさんの **4プロジェクト並行体制** :

| プロジェクト | 担当 | 状態 |
|---|---|---|
| 現行 Claude 版 MIYU(たかしさん個人用) | Aさん・Bさん | 稼働中、継続改善 |
| **汎用版「秘書召喚アプリ」(商品化目標)** | **Sさん・Tさん** | **本指示書で始動** |
| AI接客デモ「呑呑呑」(息子さん向け提案) | GPT / Codex | 別途進行中 |
| 法的書類整備(汎用版商品化向け) | エルトン | 並行整備中 |

### 1.2 「Gemini版」→「汎用版」への呼称変更

Sさん・Tさんが旧仕様書で接した可能性のある呼び名「Gemini版」は、現在は **「汎用版」** に統一されています:

- 旧: 「Gemini版」「Gemini版MIYUアプリ」
- 新: **「秘書召喚アプリ」(仮称)** または **「汎用版」**
- 理由: Gemini は AI モデルの選択肢の一つに過ぎず、プロジェクト名としては適切でない
- 将来構成: Gemini、Claude、その他の AI を抽象化して扱う(法的書類 v0.3 第11章参照)

### 1.3 現行版と汎用版の関係

- 現行版は **たかしさん個人用** として並行運用継続
- 汎用版が完成しても **現行版は引退しない**(両方並行)
- 現行版での改善ノウハウ → 汎用版へフィードバック
- 汎用版が成熟すれば、たかしさんも汎用版に乗り換える可能性あり

### 3.1 法的書類起草指示書 v0.3 から拾うべき設計影響

法的書類は「文書」だが、実は **実装に影響する論点** が混在している。Sさん・Tさんは特に以下を意識:

- **第6章 データ取り扱い**: 「運営者は通常運用において会話内容を収集・保存しない」 → 実装が「データを送らない構造」になっている必要あり
- **第6.3章 年齢制限**: 初回セットアップで年齢確認フローが必要
- **第6.7章 擬人化誤認**: AI明示宣言を起動時UIに組み込む(EU AI Act 第50条対応)
- **第11章 第三者AIサービス**: Gemini固定ではなく、AI抽象化レイヤーの設計

これらは商品化前に必須なので、初期設計時から織り込む方が手戻りが少ない。

## 4. デモ版からの継承方針(Sさん・Tさん の判断に委ねる)

たかしさんからの問い:

> ここから汎用版を作るにあたって、
> (a) デモ版をベースに利用して作るのか
> (b) そこで得た情報やノウハウだけを利用してゼロから作るのか
> どちらが良いかは、おそらくAIさんたち(開発チーム)の方がよく分かっていると思うので、そこは判断してください。

### 4.1 判断材料

**(a) デモ版ベース継承の利点・欠点**:

利点:
- Phase 0 で動くものが既に存在(立ち上がりが早い)
- たかしさんが触って確認した機能群がそのまま使える
- Calendar / Tasks の連携実装が既にある(Sさんの過去仕事)

欠点:
- デモ版は「捨てるための試作」として設計された(引継ぎ帳の哲学)
- API キー直書きなどセキュリティ的に弱い
- PWA / Capacitor 方針に合わない可能性
- ngrok 公開前提の構造が残っている

**(b) ゼロから新規構築の利点・欠点**:

利点:
- 商品化前提の堅牢な設計から始められる
- BYOK モデル(法的書類で確定)を最初から組み込める
- PWA / Capacitor + 設定画面方式を最初から織り込める
- 「未来の自分を縛らない」原則(法的書類 v0.3 第1.2)に合った設計が可能

欠点:
- 立ち上がりが遅い
- デモ版の知見を再発見する手間
- たかしさんの「動くもの」が増えるまでに時間

### 4.2 エルトンの感触(参考程度)

エルトンは(b) ゼロから新規構築 寄りの感触を持っています。理由:

- 商品化目標がある以上、最初から堅牢な構造で始める方が結局早い
- デモ版は「捨てるための試作」として設計された経緯がある
- 法的書類 v0.3 の要件(BYOK、年齢確認、AI明示、AI抽象化)はデモ版を改造して入れるより、最初から織り込む方が綺麗

ただし、これは **Sさん・Tさんの最終判断に委ねる** 。技術的な実態をエルトンより把握しているのは Sさん・Tさん。判断結果と根拠を報告してほしい。

### 4.3 判断のプロセス

1. Sさん・Tさんがデモ版コードを軽く点検
2. 法的書類 v0.3 の要件と照らし合わせる
3. 判断と根拠を Step 1 報告として提出(後述の Step 1 タスク参照)

## 8. 制約事項

### 8.1 たかしさんの意向

- **商品化目標**: いずれ App Store / Google Play 配布
- **BYOK モデル**: ユーザーが API キーを自分で持つ
- **完璧法務より誠実な説明**: 「ストア審査で落ちない最低限」を優先(法的書類 v0.3 第1.1参照)
- **未来の自分を縛らない原則**: 「○○できない」より「通常運用では○○しない」表現(法的書類 v0.3 第1.2)

### 8.2 並行プロジェクトとの関係

- 現行版(Aさん・Bさん担当)と汎用版は **別物**
- 現行版のコードを直接触らない
- 現行版から学ぶことは大歓迎
- AI接客プロジェクト(GPT / Codex 担当)は完全に別プロジェクト

### 8.3 不確実な判断は相談する

- Phase 1 の各 Step で迷ったら、独断で進めず、エルトン経由でたかしさんに確認
- 仕様書第19章「DAO 的フラット組織」の運用ルール「遠慮なしの意見交換」を守る
