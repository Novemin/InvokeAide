## 0. アプリ名について(v1.4 で新設)

### 0.1 仮称「秘書召喚アプリ」

本プロジェクトの開発中の通称は **「秘書召喚アプリ」(仮称)** とする。 これはたかしさんが本アプリの本質を「ユーザーが自分の右腕として AI 秘書を召喚し、対話を通じて日常を支援してもらう道具」と定義したことに由来する。

### 0.2 「MIYU」との関係

- **MIYU**: 本アプリに初期搭載される **キャラクター名** であり、ギャル口調・確認的・本質を突くというキャラクター性を持つ
- **秘書召喚アプリ(仮称)**: そのキャラクターを召喚する **アプリ(プラットフォーム)** の名前
- 将来は MIYU 以外のキャラクター(美輪ねえさん風、武士道おじいちゃん、ヒカル風など)も追加搭載可能(第6章参照)

つまり、 **「秘書召喚アプリ」=アプリ本体、「MIYU」=最初のキャラクター** という関係。

### 0.3 正式名称・ブランド名は別途検討

- 仮称のまま商品化しない
- 正式アプリ名・ブランド名は別途、商品化フェーズ前に確定する(v1.3 第23章 ブランド構想参照、Novem Intelligence 検討中)

## 7. 通知システム(v1.4 で軽微更新)

### 7.1 通知方式の最終採用
**Google Calendar イベントのリマインダーを通知インフラとして利用**(案C)。詳細は v1.0 / v1.1 参照。

### 7.2 通知種別の使い分け(v1.3 → v1.4 で更新)

| 通知の種類 | 使うサービス | 通知タイミング | 備考 |
|---|---|---|---|
| 朝サマリ | Google Calendar Event(秘書召喚アプリ専用カレンダー) | 5:00 | Apps Script が前夜に生成 |
| 時刻指定の予定 | Google Calendar Event(メインカレンダー) | 開始時刻 | 通常運用 |
| 時刻なし ToDo | Google Tasks(1リスト方式に変更、第17章参照) | 朝(端末標準通知) | v1.3 で 3リスト→1リスト |
| 締切リマインダー | Google Calendar Event(秘書召喚アプリ専用カレンダー) | 3日前から | ToDo から自動生成 |

### 7.3 専用カレンダーの自動作成
ユーザーのメインカレンダーが予定で埋め尽くされないように、本アプリは初回起動時に専用のサブカレンダー(名前: "MIYU" など)を自動作成する。

### 7.4 朝サマリの Calendar イベント本文
要約3〜5行 + 「📱 アプリで詳細を聞く」リンクのハイブリッド方式を採用。

### 7.5 通知の確実性確保
イベント作成時に `reminders` を明示的に設定し、`overrides: [{ method: 'popup', minutes: 0 }]` で確実に通知。

## 8. Google Calendar API 連携(v1.4 で新設、具体化)

本アプリで **Google Calendar API ネイティブでサポートされている機能のうち、秘書としての日常利用に必要なものをすべて実装する** 。各機能を1項目ずつ具体的に列挙し、「何をするか」「何ができるようになるか」を明記する。

### 8.1 実装する機能(必須)

#### 8.1.1 イベントの作成(create_event)

- **何をするか**: ユーザーの発話から Calendar 上に新しい予定を作成する
- **何ができるようになるか**:
  - 「明日17時から19時、銀座アイブリューで友達と食事」と一言で予定が入る
  - 開始時刻・終了時刻・タイトル・場所・メモ・繰り返しを一括で指定できる
- **API パラメータと挙動**:

| パラメータ | 必須 | 用途 | 認識キーワード例 |
|---|---|---|---|
| summary(タイトル) | ✅ | 予定名 | 自動抽出 |
| start.dateTime | ✅ | 開始日時 | 「○時から」「○月○日」 |
| end.dateTime | ✅ | 終了日時 | 「○時まで」/ 未指定なら +1時間 |
| location | ⬜ | 場所 | 「場所」「会場」「住所」「@◯◯」 |
| description | ⬜ | メモ・詳細 | 「メモ」「詳細」「内容」 |
| recurrence | ⬜ | 繰り返し | 「毎週」「毎月」「毎日」 |
| reminders | ⬜ | 通知設定 | デフォルトは popup 0分前 |

#### 8.1.2 イベントの取得・検索(list_events / search_events)

- **何をするか**: ユーザーの問い合わせに基づき、Calendar から予定を検索して返す
- **何ができるようになるか**:
  - 「今日の予定教えて」「来週の予定は?」「銀座アイブリューの予定いつだっけ?」に MIYU が答えられる
  - 時間範囲指定(今日 / 明日 / 今週 / 来週 / 特定日)と、タイトル文字列での検索の両方をサポート
- **取得時に返すフィールド**: summary, start.dateTime, end.dateTime, location, description, id

#### 8.1.3 イベントの更新(update_event)

- **何をするか**: 既存の予定の任意のフィールドを変更する
- **何ができるようになるか**:
  - 「○○の予定の時間を1時間ずらして」→ start.dateTime / end.dateTime 更新
  - 「○○に場所追加して、新宿駅で」→ location 上書き
  - 「○○のメモに『〇〇』を追加しといて」→ description 追記(末尾にスペース挟む)
  - 「○○のメモ全部書き直して。『〇〇』だけにして」→ description 上書き
- **重要: description の追記/上書き両対応**(v1.4 で追加):
  - **append_description**: 既存内容の末尾にスペースを挟んで追記。履歴保護用、デフォルトはこちら
  - **new_description**: 既存内容を完全に置き換え。「書き直し」「変更」「上書き」キーワードで使用
  - 両方指定された場合は new_description を優先、append_description は無視
  - 認識ロジック:
    - 「追加」「追記」「足して」「補足」→ append_description
    - 「書き直し」「変更して」「上書き」「全部入れ替え」→ new_description
    - 迷ったら append を選択(履歴保護優先)
- **挙動の原則**: location は上書き、description は追記がデフォルト

#### 8.1.4 イベントの削除(delete_event)

- **何をするか**: 既存の予定を削除する
- **何ができるようになるか**:
  - 「○○の予定キャンセルして」「○○削除して」で予定が消える
- **安全策(v1.3 第24章 確認設計の原則に基づく)**:
  - 削除実行前に必ず3択ボタン式で確認: ①お願いします / ②やらない / ③保留
  - 「やらない」を選んだら処理中止
  - 「保留」を選んだら ToDo に「○○の予定を削除するか検討」を追加

### 8.2 実装する機能(便利機能)

#### 8.2.1 繰り返し設定(recurrence)

- **何をするか**: RRULE 形式の繰り返しルールを設定する
- **何ができるようになるか**:
  - 「毎週月曜の10時から定例MTG入れて」→ `["RRULE:FREQ=WEEKLY;BYDAY=MO"]`
  - 「毎日朝5時に朝サマリ」→ `["RRULE:FREQ=DAILY"]`
  - 「毎月25日に給料日メモ」→ `["RRULE:FREQ=MONTHLY;BYMONTHDAY=25"]`
- **対応するパターン例**:
  - 毎日 / 平日のみ / 毎週○曜 / 隔週 / 毎月○日 / 毎年○月○日
- **制約**: 複雑な例外(「毎週月曜だが祝日は除く」など)は MVP では対応しない、Phase 4 以降検討

#### 8.2.2 参加者の招待(attendees)

- **何をするか**: 予定にメールアドレス指定で参加者を招待する
- **何ができるようになるか**:
  - 「明日のMTG、田中さん(tanaka@example.com)も招待しといて」
- **MVP では除外、Phase 3 以降検討**: 個人秘書としては優先度が低く、また「人物データベース」との連携設計が必要

#### 8.2.3 リマインダー設定の個別調整(reminders)

- **何をするか**: 予定ごとの通知タイミングを変更する
- **何ができるようになるか**:
  - 「30分前に通知して」→ `{ method: 'popup', minutes: 30 }`
  - 「前日にも通知ほしい」→ `{ method: 'popup', minutes: 1440 }`

#### 8.2.4 色分け(colorId)

- **何をするか**: 予定に色を付ける
- **何ができるようになるか**:
  - 仕事系は青、プライベートは赤、など視覚的に整理
- **MVP では除外、Phase 4 以降検討**: 個人秘書としては優先度が低い

#### 8.2.5 ビデオ会議リンクの自動生成(conferenceData)

- **何をするか**: Google Meet の URL を自動生成して予定に紐付ける
- **何ができるようになるか**:
  - 「明日のMTG、オンラインで」→ Meet リンク付きの予定が作成される
- **MVP では除外、Phase 4 以降検討**

#### 8.2.6 添付ファイル(attachments)

- **何をするか**: Google Drive のファイルを予定に添付する
- **MVP では除外、Phase 4 以降検討**

### 8.3 v1.4 で確定したルーティング・キーワード

以下のキーワードを `routeToolsFor` の判定ロジックに組み込む:

- **検索系キーワード(grounding 経路を最優先)**: 天気 / 気温 / 気象 / 天候 / ニュース / 最新 / 株価 / 為替 / レート / 時刻 / 時差
- **Calendar 操作キーワード(functions 経路)**: 予定 / スケジュール / カレンダー / アポ / 会議 / MTG
- **時間表現(functions 経路、ただし検索系キーワードが優先)**: 今日 / 明日 / 明後日 / 来週 / 今週 / ○月○日

判定順序:
1. 検索系キーワード → grounding 即決
2. functions 系キーワード → functions
3. 日付正規表現 → functions
4. デフォルト → grounding

## 9. Google Tasks API 連携(v1.4 で新設、具体化)

Google Tasks API はシンプルな ToDo 管理用に設計されており、 **Calendar に比べてできることが大幅に少ない** 。本仕様では「API で技術的に可能な機能はすべて実装する」「不可能な機能は明示的に記録する」両方を行う。

### 9.1 実装する機能(API 上可能なもの)

#### 9.1.1 タスクの作成(create_task)

- **何をするか**: ユーザーの発話から Tasks に新しいタスクを追加する
- **何ができるようになるか**:
  - 「部屋を片付けるってタスクに入れて」「明日までに資料作るってタスク」と一言で追加
  - タイトル・期限日(時刻なし)・メモ(notes)・親タスク(parent)を指定可能
- **API パラメータと挙動**:

| パラメータ | 必須 | 用途 |
|---|---|---|
| title | ✅ | タスク名 |
| due(日付のみ、時刻不可) | ⬜ | 期限日(YYYY-MM-DD) |
| notes | ⬜ | メモ |
| parent | ⬜ | 親タスクの ID(サブタスク化用) |

#### 9.1.2 タスクの取得・検索(list_tasks)

- **何をするか**: タスクリストから未完了/完了状態でタスクを取得
- **何ができるようになるか**:
  - 「今のタスク全部教えて」「今日締切のタスクある?」に答えられる
- **取得時に返すフィールド**: title, due, notes, status, completed, parent, id

#### 9.1.3 タスクの更新(update_task)

- **何をするか**: 既存タスクのタイトル・期限・メモを変更する
- **何ができるようになるか**:
  - 「○○のタスクを明日締切に変更して」
  - 「○○のメモ追加して、『〇〇』」

#### 9.1.4 タスクの完了(complete_task)(v1.4 で新設)

- **何をするか**: タスクの status を "completed" に変更する
- **何ができるようになるか**:
  - 「○○のタスク完了にして」「○○終わった」で完了マーク
  - 完了日時(completed フィールド)も自動記録
- **認識キーワード**: 「完了」「終わった」「やった」「片付いた」「クリア」

#### 9.1.5 タスクの削除(delete_task)

- **何をするか**: タスクを削除する(soft delete / hard delete 両対応)
- **何ができるようになるか**:
  - 「○○のタスク削除」「○○いらない、消して」
- **安全策**: Calendar 削除と同様、3択ボタン式で確認

#### 9.1.6 サブタスク(階層化、parent)

- **何をするか**: タスクの親子関係を構築する
- **何ができるようになるか**:
  - 「『引越し準備』の下に『段ボール調達』『電気手続き』を入れて」
- **MVP では除外、Phase 3 以降検討**: 個人秘書としての複雑度が高い、ユーザー需要次第

#### 9.1.7 タスクの順序変更(position)

- **何をするか**: リスト内のタスクの並び順を変える
- **MVP では除外、Phase 4 以降検討**

### 9.2 実装しない機能(API 仕様上不可能なもの)

以下は **Google Tasks API の仕様上、技術的に実装が不可能** であるため、本アプリでは Tasks に対しては実装しない。秘書召喚アプリでこれらの機能が必要な場合は、 **Calendar 側で実装** する(運用方針は第17章参照)。

| 機能 | 不可能の理由 | 代替策 |
|---|---|---|
| 時刻指定(○時○分) | API 仕様で due の時刻部分は自動破棄される(公式ドキュメント明記) | 時刻ありは Calendar に登録 |
| 場所(location) | Tasks リソースに location フィールド自体が存在しない | 場所が必要な ToDo は Calendar に登録、または notes に文字列で記載 |
| 繰り返し(recurrence) | API では繰り返しルールの読み書き不可(スマホアプリでは UI 上設定できるが、API からは見えない) | 繰り返しが必要な ToDo は Calendar の繰り返し予定として登録 |
| 参加者の招待 | Tasks は1人専用ツール、共有・招待概念なし | 共有が必要なら Calendar に登録 |

### 9.3 「時刻あり = Calendar、時刻なし = Tasks」の運用原則

v1.2 から継承し、v1.3 で 1リスト方式に統一(第17章)、v1.4 で API ベースの裏付けを明記。

- **時刻が決まっている**(○時から〜) → Calendar Event として登録
- **時刻未定、日付のみ**(○月○日まで) → Tasks に due 付きで登録
- **日付すら未定**(いつかやる) → Tasks に due なしで登録
- **機が熟したら昇格**: Tasks 上のタスクで時刻が決まったら、Calendar Event に移行(MIYU が促進ではなく抑制、ユーザー判断、v1.3 第17章参照)

## 17. ToDo 運用(v1.3 で大幅改訂、v1.4 で軽微更新)

### 17.1 1リスト方式への転換(v1.3)

v1.2 の3リスト方式(未完了 / 完了 / キャンセル)から、 **1リスト方式** に変更:
- Google Tasks に `ToDo_by_MIYU` という1つのリスト
- 状態管理: 未完了=☐、完了=☑、キャンセル=削除(事前確認あり)

### 17.2 ランク機能(v1.3 で確定)

A/B/C/D/**P** ランクの扱い:
- A: 緊急かつ重要
- B: 緊急でないが重要
- C: 緊急だが重要でない
- D: 緊急でも重要でもない
- P: Purchase(購入・買い物)

### 17.3 期限切れ時の処理

A/B ランクのタスクが期限切れになった場合:
- ユーザーに「延期 / 保留 / 削除」を3択ボタンで確認
- 削除実行前には必ず再確認

C/D ランクは入力日 + 7日経過で日曜夕方に一括削除(事後報告のみ)

### 17.4 ランク自動付与・昇格判断

- **新規 ToDo のランク設定**: MIYU が自動付与
- **昇格判断**(時刻なし Tasks → 時刻あり Calendar): ユーザー判断、MIYU は抑制側
- 週1レビュー時に「機が熟したか」を確認、無理な日程入れはプレッシャー化するため禁止

### 17.5 デュアル運用方針

- **一般ユーザー**: 最初から Tasks 1リストに直接追加

## 26. 配信・配布インフラ(v1.4 で新設)

本アプリは「ユーザーの Google アカウント内で完結するサーバーレス構成」を目指すが、 **開発・テスト段階および将来の商品化段階で、配信方法を段階的に選択する必要がある** 。本章では各段階で採用する配信方法を明記する。

### 26.1 配信方法の選択肢

| 方法 | 用途 | メリット | デメリット |
|---|---|---|---|
| ① ローカルファイル直接(file://) | 開発初期、PC内テスト | 設定不要 | iOS Safari で動作不可、スマホで使えない |
| ② ngrok 一時公開 | デモ・少人数テスト | 数分でスマホアクセス可能、PC が起点 | URL が毎回変わる(無料版)、PC 必要 |
| ③ Cloudflare Pages / Vercel | 内輪ベータ・限定公開 | 永続 URL、無料、スマホ常用可 | APIキー保護必須、HTTPS 化 |
| ④ PWA(Progressive Web App) | 一般ベータ・初期商品版 | App Store 不要、URL 共有で配布、オフライン対応可 | iOS で機能制限あり |
| ⑤ ハイブリッドアプリ(React Native 等) | 本格商品化 | App Store 配布、フル機能 | 開発コスト大、Mac 必須(iOS) |

### 26.2 各 Phase での採用方針

| Phase | 主な配信方法 | 補助 |
|---|---|---|
| Phase 0(2026-05-16 達成) | ② ngrok 一時公開 | ① PC ローカル開発 |
| Phase 1〜3 | ② ngrok or ③ Cloudflare Pages | 機能拡張優先 |
| ベータ版配布(Urban Hippies 内輪) | ④ PWA | ③ も併用検討 |
| 正規版商品化 | ④ PWA + ⑤ React Native 検討 | App Store 申請も視野 |

### 26.3 ngrok 使用時のセキュリティ対策(v1.4 で確定)

ngrok で公開する HTML ファイルには Gemini API キー / OAuth client_secret / refresh_token がハードコードされている場合がある。第三者にトンネル URL が漏れて DevTools で覗かれると、これらの認証情報が露出する。

#### 26.3.1 基本対策(必須)

- **Basic 認証を必ず付与**: `ngrok http --basic-auth "user:pass" 8080`
  - ユーザー名・パスワードは最低8文字以上
  - パスワードは口頭でのみ共有、メール・SNS では送らない

#### 26.3.2 セッション管理(必須)

- **デモ後は ngrok セッションを必ず止める**: 長時間放置しない
- **PC を持ち運ぶ際**: スリープ復帰時に ngrok URL は変わるため、再起動と再配布が必要

#### 26.3.3 認証情報のローテーション(推奨)

- デモ後は **OAuth refresh_token を必ずローテート**:
  - Google Cloud Console から旧トークンを失効、新規発行
  - HTML 内のハードコード値も更新
- API キーも定期的にローテート(月1回など)

### 26.4 配信時の依存ソフトウェア(Phase 0 実績)

ngrok ベースのデモ環境で実証済みの構成:

| 用途 | ソフトウェア | バージョン |
|---|---|---|
| ローカル HTTP サーバー | `npx http-server` | 14.1.1(2026-05-16 時点) |
| トンネル | `ngrok` | 3.39.2(無料プラン、要 3.20 以上) |
| ローカル前提環境 | Node.js | v24.15.0 |
| 開発マシン OS | Windows 11 | - |
| ターゲットデバイス | iPhone Safari | iOS 17 以降 |

### 26.5 配信時の運用ハマりポイント(v1.4 で記録)

Phase 0(2026-05-16)で実際に発生・解消したトラブル:

1. **ngrok バージョン不一致**: アカウント要求バージョン(3.20+) と winget インストール版(3.3.1)の不一致
   - 対処: 公式サイトから手動 ZIP DL → ローカルフォルダに配置 → 実行
2. **フォルダ名のスペース問題**: `C:\dev\MIYU Demo` のスペースで cd が失敗
   - 対処: PowerShell でパスをダブルクォートで囲む(`cd "C:\dev\MIYU Demo"`)
3. **2台 PC 間のファイル転送**: 開発機(PC2)と配信機(PC1)で別フォルダ
   - 対処: Google Drive のマイドライブ経由(共有リンク生成は厳禁)
4. **HTML 内認証情報の露出リスク**: ngrok URL から DevTools で API キー丸見え
   - 対処: 26.3 のセキュリティ対策(Basic 認証 + 事後ローテーション)

これらは将来 PWA / Cloudflare Pages 移行時にも参考になる経験則として記録する。
