## 1. 起草指示書 v0.3 から抽出した主要論点（指示文 §3-1）

本起草で**必ず守るべきポイント**を起草指示書 v0.3 から抜き出して整理する。文言は原文を改変せず引用する。

### 1-1. 第1章「未来の自分を縛らない」原則（§1.2）

> 「断定表現を避ける」: 「存在しない」「一切〜しない」のような絶対的表現は、将来の機能追加（クラッシュ分析、エラー収集、Telegram 連携、Push 通知、ライセンス認証等）で矛盾を生む
> 「限定表現を使う」: 「専用サーバーを運用していない」「通常運用において収集しない」など、**何を約束していないかが明確に分かる**表現
> これは「曖昧にする」ことではない。「正確に約束する」ことである。

### 1-2. 第6章「データの取り扱い」表現方針（§6.5）

統合型ゆえ「ここがプライバシーポリシー本体となる」と明記（§6.5）。**「未来の自分を縛らない」原則を厳格に適用する**。

#### §6.5.1 採用すべき表現（原文ママ）

| v0.2 までの表現（避ける） | v0.3 で採用する表現 |
|---|---|
| 「運営者側サーバーは存在しない」 | 「運営者は、ユーザーの会話内容を保存・閲覧することを目的とした専用サーバーを運用していません」 |
| 「ユーザーデータは一切受信・保存しない」 | 「運営者は、通常運用において、ユーザーの会話内容を収集・保存しません」 |
| 「ユーザーデータにアクセスできない」 | 「運営者は、通常運用において、ユーザーの会話内容にアクセスする手段を持ちません」 |

#### §6.5.2 起草時に含める要素

- ユーザーデータの保管場所: **ユーザー自身の Google Drive、端末内**
- Gemini API への通信内容には Google の利用規約・プライバシーポリシーが適用される旨
- API キーは端末内に暗号化保存される
- **将来追加される可能性のあるデータ収集**（クラッシュレポート / 利用統計 / ライセンス認証）について明示的に余地を残す。導入時は**規約改訂と事前通知を経る**ことを明記

### 1-3. 表現の置換ルール（§8.9 の表、原文ママ）

| 避ける表現 | 推奨表現 |
|---|---|
| サーバーは存在しない | 専用サーバーを運用していない |
| データを一切受信しない | 通常運用において収集・保存しない |
| 〜にアクセスできない | 〜にアクセスする手段を持たない |
| 100% 安全 | 合理的なセキュリティ対策を講じる |
| 永久に〜しない | 現時点では〜していない |

### 1-4. 限定表現のキーワード

起草で多用する定型句:「**目的とした**」「**通常運用において**」「**現時点では**」「**合理的な**」「**運営者の故意または重過失による場合を除き**」（§8.8 消費者契約法第8条・第8条の2 対応）。

### 1-5. その他、本起草で関係する論点

- **AI 明示宣言**（§6.1、EU AI Act 第50条）: プライバシーポリシー単体でも「本アプリは AI を用いた対話システムです」を含める
- **年齢制限**（§6.3）: 13歳未満不可、13〜18歳未満は保護者同意
- **第三者AIサービス抽象化**（§6.9）: Gemini 固定でなく Claude / OpenAI 等への変更可能性を限定表現で予告
- **BYOK のデータ非保有構造**（§8.1）: 「該当しない」と断言せず「データを受信しない仕組みである」ことを明示

## 2. Google OAuth 審査の要件（指示文 §3-2、引用元URL付き）

> **凡例**: 引用は Google 公式ドキュメントの英語原文を該当箇所のみ抜粋。日本語は要約。**本起草時に技術顧問が下記URLを再確認すること**（指示文 §6）。

### 2-1. プライバシーポリシーに何を書くべきか（必須）

公式要件（OAuth App Verification）:

> "disclose the manner in which your application accesses, uses, stores, or shares Google user data"
> （アプリが Google ユーザーデータに**アクセス・利用・保存・共有する方法**を開示すること）

出典: [Sensitive scope verification | Google for Developers](https://developers.google.com/identity/protocols/oauth2/production-readiness/sensitive-scope-verification)、[OAuth App Verification Requirements (support.google.com/cloud/answer/13464321)](https://support.google.com/cloud/answer/13464321)

→ **各 OAuth スコープごとに「何のデータに、何の目的でアクセスし、どこに保存するか」を明示する必要がある**。これが本ポリシーの中核。

### 2-2. 公開URLが満たすべき条件

公式要件（support.google.com/cloud/answer/13464321 より）:

> "The Privacy Policy should be hosted within the domain that hosts your homepage."
> （プライバシーポリシーは**ホームページと同一ドメイン**でホストすること）
> "linked on your homepage so that users can find this information easily"
> （**ホームページからリンク**され、ユーザーが容易に見つけられること）
> "linked from the OAuth consent screen on the Google API Console"
> （**OAuth 同意画面からもリンク**されること）
> "The Privacy Policy linked on your homepage and the one linked on the app's OAuth consent screen should be same"
> （ホームページと同意画面のプライバシーポリシーは**同一であること**）

ホームページ側の要件:

> "hosted on a verified domain you own" / "describe your app's functionality to its users"（cannot be login-only）
> （**所有を証明した検証済みドメイン**上にあり、アプリの機能を説明していること。ログイン画面だけは不可）

ドメイン所有証明:

> "verify ownership of all domains ... using Google Search Console"
> （**Google Search Console** で全ドメインの所有を証明）

→ InvokeAide では **club-freedom.tokyo の HTTPS 化（引き継ぎ帳 §1-2 の宿題）が前提**。公開先は同ドメイン配下（例 `https://club-freedom.tokyo/invokeaide/privacy/`、要確認 §4-Q1）。

### 2-3. センシティブ / 制限付きスコープに対する個別要件

スコープは **非センシティブ / センシティブ / 制限付き（restricted）** の3分類。InvokeAide の各スコープ:

| スコープ | 分類 | 根拠（出典） |
|---|---|---|
| `drive.file` | **非センシティブ** | "The `drive.file` scope is non-sensitive ... it allows for a more streamlined verification process" — [Choose Google Drive API scopes](https://developers.google.com/workspace/drive/api/guides/api-specific-auth) |
| `tasks` | **センシティブ**（※公式リストで最終確認推奨） | センシティブ/制限付きの広域カテゴリに含まれる。[OAuth scopes reference](https://developers.google.com/identity/protocols/oauth2/scopes) で要最終確認 |
| `calendar` / `calendar.events` | **センシティブ** | "Examples of sensitive scopes include reading events stored in Google Calendar" — [scope classification 検索結果](https://developers.google.com/identity/protocols/oauth2/scopes) |

**重要な結論**:
- InvokeAide は **制限付き（restricted）スコープを使わない**（フルの `drive` を避け `drive.file` を採用しているため）。
- したがって**制限付きスコープに課される「年次セキュリティ評価（security assessment）」は不要**。

> 制限付きスコープの追加要件（InvokeAide には**非該当**だが参考）:
> "Security assessments (annual)" / "Demonstration videos" / "Strict data limitation rules prohibiting transfers to third parties, advertisers, or data brokers"
> 出典: [support.google.com/cloud/answer/13464321](https://support.google.com/cloud/answer/13464321)

- センシティブスコープ審査で必要なもの:
  > "Request narrowest scopes necessary" / "Verify domain ownership" / "Maintain current contact information" / "Submit detailed justifications and demonstration videos"
  > 出典: [Sensitive scope verification](https://developers.google.com/identity/protocols/oauth2/production-readiness/sensitive-scope-verification)
  → **デモ動画**と**各スコープの利用理由（justification）**は OAuth 申請時に必要（プライバシーポリシー本体とは別だが、整合させる）。

### 2-4. 各スコープの「利用目的説明」をどこに書くか

公式は「プライバシーポリシー内で Google ユーザーデータの access/use/store/share を開示」と求める（§2-1）。
→ プライバシーポリシー内に **「取得する情報」+「利用目的（スコープごと）」セクションを設け、スコープ単位で1対1に記述**するのが素直。OAuth 同意画面の justification と文言を揃える。

### 2-5. Limited Use（限定利用）開示要件

Google API Services User Data Policy の **Limited Use** 要件。公式の推奨開示文:

> "{App's} use of information received from Google APIs will adhere to Google API Services User Data Policy, including the Limited Use requirements."
> 出典: [support.google.com/cloud/answer/13463817](https://support.google.com/cloud/answer/13463817)（Limited Use の項）

→ **プライバシーポリシーに「Google API から受け取った情報の利用は Google API Services User Data Policy（Limited Use 要件を含む）に従う」旨の一文を入れる**のが定石。特に AI/ML 利用がある場合（InvokeAide は Gemini 連携あり）に開示が重視される。

> **要確認（§4-Q9）**: InvokeAide は Drive/Calendar/Tasks のデータを Gemini に渡す場面があるか（例: 予定を読んで要約）。ある場合、Limited Use と AI 利用の関係を本文で丁寧に書く必要。本書では事実確認に留め、判断は保留。

## 3. 既存資料との突合結果・不整合点（指示文 §3-3）

突合対象: 起草指示書 v0.3 / 仕様書 v1.4・v1.2 §15 / 思想書 v0.1 / 既存コード（`src/interfaces/AuthProvider.ts` 等）/ Phase2 設計（OAuth スコープ設計 v0.1・Drive レイアウト v0.1）/ CLAUDE.md §5.1。

### 不整合 ①【最重要】Calendar スコープが `calendar`（フル）か `calendar.events` か矛盾

| 資料 | 記述 |
|---|---|
| CLAUDE.md §5.1（2026-05-28 確定） | OAuth スコープ = `calendar.events`、`tasks`、`drive.file`（制限付きスコープなし） |
| 本指示文 §3-2 | 「InvokeAide は `calendar.events` `tasks` `drive.file` を使用」 |
| **Uさん OAuth スコープ設計 v0.1 §2.1** | **`calendar`（フル）が必要**。理由:「専用サブカレンダー作成には `calendar` フルスコープが必要（`calendar.events` は既存カレンダー内 Event のみ操作可）」 |
| `src/interfaces/AuthProvider.ts` L52-53 | `AuthConfig.stage2AdditionalScopes` の例コメントが `['calendar']`（フル） |

**何が問題か**: プライバシーポリシーの「利用目的」には**実際に要求するスコープ**を正確に書く必要がある（§2-4）。`calendar` フルと `calendar.events` では開示文言が変わる。さらに、もし `calendar.events` に確定するなら、Uさん 設計の「専用カレンダー自動作成（メインカレンダーを汚染しない）」が**技術的に成立しない**可能性があり、仕様書 v1.4 §7.3 の設計意図とも衝突する。
**分類への影響**: どちらもセンシティブ（制限付きではない）なので**審査の重さは変わらない**。違いは①開示文言、②専用カレンダー機能の成否。
**対応**: §4-Q5 でたかしさんに確定を仰ぐ。本起草はその確定後。

### 不整合 ②【表現方針】仕様書 v1.2 §15 が断定表現のまま（起草指示書 v0.3 と逆行）

仕様書 v1.2（v1.4 が §15 を継承）の該当箇所:

- L372「たかしさん側のサーバー: **存在しない**」
- L389「MIYU は会話内容を**一切受信・保存しない**」
- L390「API キーは端末内に暗号化保存され、外部送信されない」

これらは起草指示書 v0.3 §6.5.1 / §8.9 が**明確に「避ける」と指定した断定表現**。
**対応**: プライバシーポリシー起草時は v1.2 §15 の文言を**そのまま流用せず**、必ず §6.5.1 の限定表現に置換する。仕様書本体の改訂は本起草の範囲外（エルトン主導の v1.5 反映候補として申し送り）。**ポリシー側が正、仕様書側は将来追従**という関係を明記しておく。

### 不整合 ③【データ取扱】「会話ログを保存」と「通常運用で収集しない」の整合

指示文 §3-3 が例示した論点。突合結果は**「矛盾ではないが、書き分けが必須」**。

- 仕様書 / Uさん Drive レイアウト設計 v0.1 §3.1: 会話ログは `MIYU_App_Data/logs/conversations/会話ログ_YYYY-MM-DD.md` に**保存される**。設定・プロファイルも Drive に保存。
- 起草指示書 v0.3 §6.5: 「運営者は、通常運用において、ユーザーの会話内容を収集・保存しません」。

**両立の鍵（事実整理、法的判断はしない）**: 保存先は**ユーザー自身の Google Drive と端末内**であり、**運営者のサーバーではない**。「保存される（ユーザーの領域に）」と「運営者が収集・保存しない」は主語が異なる。
**対応**: プライバシーポリシーで**保存の主体と場所を明確に分離して記述**する。
- 「アプリは、ユーザーの設定・会話履歴等を**ユーザー自身の Google Drive および端末内**に保存します」
- 「運営者は、通常運用において、これらの内容を収集・保存しません」
両文を併記してはじめて誠実かつ正確になる（Uさん Drive 設計 §0.2 原則2「Drive は SoT、運営者アクセス無し」と整合）。

### 不整合 ④【スコープ十分性】`drive.file` だけで「Drive 完結」が足りるか

指示文 §3-3 が例示した論点。突合結果は**「足りる。設計上の意図的選択」**。

- Uさん OAuth 設計 v0.1 §4 が `drive.file` を**本命として明示選定**。理由: 最小権限・ユーザー透明性・`MIYU_App_Data/` がユーザーに可視。
- `drive.file` は「アプリが作成 / ユーザーが選択したファイルのみ」にアクセス可。InvokeAide は自分で `MIYU_App_Data/` を作るため**ユーザー操作なしで完結**（OAuth 設計 §4.4）。
- 公式（§2-3）でも `drive.file` は非センシティブで「streamlined verification」。

**結論**: `drive.file` で足りる。むしろ**フルの `drive` を避けたことで制限付きスコープ審査（年次セキュリティ評価）を回避できている**のが設計上の利点。プライバシーポリシーでもこの「最小権限」を積極的に記述するとよい（審査官への誠実なアピールにもなる）。

### 整合確認できた点（参考）

- **思想書 v0.1**「人間味のための AI」「効率化のための AI ではない」「召喚」コンセプト ↔ `drive.file` の透明性選定（OAuth 設計 §4.2）は哲学的に整合。プライバシーポリシーの「はじめに」でこのトーンを反映可能。
- **API キー（BYOK）は IndexedDB 端末内 AES-GCM 暗号化**（Uさん Drive 設計 §3.2、OAuth 設計 §5）↔ 起草指示書 §6.5.2「API キーは端末内に暗号化保存」と整合。
- **リフレッシュトークンは Drive に置かず端末内**（OAuth 設計 §5.2）↔ 「認証情報は端末ローカル」と一貫。

## 5. 起草の構成案（章立て）の提案（指示文 §3-5）

OAuth 審査用プライバシーポリシー**単体**として最低限必要な章立て。起草指示書 v0.3 第5章（15章・統合版）から**プライバシー関連章を抽出・再構成**した、シンプルかつ Google 要件（§2-1〜2-5）を満たす構成。

```
表題: 秘書召喚アプリ（InvokeAide）プライバシーポリシー
      バージョン / 施行日（Q6）

1. はじめに
   - 本アプリは AI を用いた対話システムである旨（AI明示宣言、起草指示書 §6.1 / EU AI Act 50条）
   - 「Novem Intelligence（以下「運営者」）」の定義（起草指示書 §2.2）
   - 本ポリシーの対象範囲

2. 基本方針（BYOK・データ非保有構造）
   - データはユーザー自身の Google Drive と端末内に保管される旨（§3 不整合③の書き分け）
   - 運営者は通常運用において会話内容を収集・保存しない旨（限定表現、§6.5.1）
   - 「専用サーバーを運用していない」（断定回避、§8.9）

3. 取得する情報と Google ユーザーデータへのアクセス
   - Google アカウント情報（openid / email / profile）
   - Google Drive（drive.file: アプリが作成したファイルのみ／最小権限）
   - Google ToDo（tasks）
   - Google カレンダー（calendar or calendar.events ← Q5 確定後）
   - API キー（BYOK、端末内 AES-GCM 暗号化）

4. 利用目的（OAuthスコープごとに1対1で明示）★Google要件の核心（§2-1, §2-4）
   - スコープ × 目的 × 保存場所 を表形式で
   - 同意画面の justification と文言を一致させる

5. 第三者提供・第三者サービス
   - Gemini API（Google）への送信内容には Google の規約が適用される旨（§6.5.2）
   - 第三者AIサービス（Claude/OpenAI 等）への将来変更可能性（限定表現、§6.9）
   - Limited Use 遵守の一文（§2-5）★Google要件

6. 保存場所・期間とデータの所在
   - Drive（MIYU_App_Data/ 配下）/ 端末内 IndexedDB の役割分担
   - 認証情報（リフレッシュトークン）は端末内のみ・Drive 非保存

7. ユーザーの権利（閲覧・削除・撤回）
   - Drive 内データの自己削除手順（起草指示書 §6.10）
   - OAuth 連携解除（Google アカウント側からの取消）
   - アプリ削除と API キー失効

8. セキュリティ対策
   - 「合理的なセキュリティ対策を講じる」（断定回避、§8.9）
   - 端末内暗号化、最小権限スコープ

9. 年齢制限
   - 13歳未満不可、13〜18歳未満は保護者同意（起草指示書 §6.3）

10. 改訂方針
    - 限定表現での将来データ収集予告（§6.5.2）
    - 改訂時の通知方法・バージョン管理（Q6 連動）

11. お問い合わせ
    - 連絡先（Q2）、運営者表記（Q3）

12. 準拠法（任意・統合版に合わせる場合）
    - 日本法（起草指示書 §6.12）。単体ポリシーでは省略可
```

**構成上の Tさん 所見**:
- Google が最重視するのは **第3章（取得情報）+ 第4章（スコープ別利用目的）+ 第5章（Limited Use）**。ここを厚く、スコープと1対1で書くこと。
- 第2章で「データ非保有構造」を**限定表現**で誠実に書くことが、起草指示書 v0.3 の哲学と Google 審査の両方に効く。
- 統合版（規約＋ポリシー＋免責）とは別に、**この単体ポリシーを独立ページとして公開**するのが審査上わかりやすい（Q4 と連動）。
