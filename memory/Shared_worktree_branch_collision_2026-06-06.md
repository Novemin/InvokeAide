# 共有ワークツリーでのブランチ衝突ハザード（S/T/U 並行作業）

**記録**: 2026-06-06、Sさん（S2-1 実装中に実際に遭遇・解決）。

## 何が起きるか（事象）

S・T・U は同一ワークツリー `C:\dev\InvokeAide` を共有している。git の HEAD（チェックアウト中ブランチ）はワークツリーに1つだけなので、**あるエージェントが `git checkout` すると、他エージェントの作業ブランチも同時に切り替わる**。

- 2026-06-06 実例: Sさんが composition root をビルド確認→コミットする間に、HEAD が Uさんの `feature/voicevox-tts-provider` に切り替わっており、Sさんのコミットが誤って Uさんのブランチに乗った。
- 引き継ぎ帳 2026-05-30 §8-2「同時並行作業のキューイング問題」・§7-4「並行コミット問題」と同根。

## なぜ起きるか（Why）

git のワークツリーは HEAD を1本しか持てない。複数エージェントが同じディレクトリで別ブランチを触ると、最後に checkout した者の状態に全員が引きずられる。

## どう対処するか（How to apply）

1. **各コミット直前に必ず `git branch --show-current` で意図したブランチか検証する**（Sさん は S2-1 で全コミットこれを徹底し、以降の汚染ゼロ）。
2. 汚染が起きたら慌てず：自分のコミットを正しいブランチに `cherry-pick` → 元ブランチを相手の正しい先端に `git branch -f <branch> <sha>` で戻す。**reflog が残るのでデータ損失なし**。
3. 構造的に排除したい場合は `git worktree add` で各自専用ディレクトリに分離する案がある（2026-06-06 たかしさん判断: 当面は「共有ツリーで続行・コミット毎に検証」を選択）。

## 補足: Bash ツールでのコミットメッセージ

PowerShell here-string `@'...'@` を **Bash ツールで使うと `@` がメッセージに混入する**。Bash では `-m` を複数指定して段落を分ける方式に統一すること。
